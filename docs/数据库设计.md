## 二、详细字段说明

### 2.1 users

```sql
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    username        VARCHAR(32)  NOT NULL UNIQUE,
    password_hash   CHAR(32)     NOT NULL,    -- md5 32 hex
    salt            CHAR(24)     NOT NULL,    -- 12 bytes -> 24 hex
    real_name       VARCHAR(32)  NOT NULL,
    employee_no     VARCHAR(16)  NOT NULL UNIQUE,
    gender          gender_type  NOT NULL DEFAULT 'other',
    age             SMALLINT,
    role            user_role    NOT NULL DEFAULT 'admin',
    is_active       BOOLEAN      NOT NULL DEFAULT TRUE,
    last_login_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),

    CHECK (age IS NULL OR age BETWEEN 16 AND 100)
);
```

**索引**:
- `users.role` — RBAC 查询过滤
- `users.is_active WHERE is_active` — 部分索引,只索引启用账号

### 2.2 books

### 2.3 purchase_orders (四态机)

```sql
CREATE TYPE purchase_status AS ENUM ('pending', 'paid', 'returned', 'received');

CREATE TABLE purchase_orders (
    id            BIGSERIAL PRIMARY KEY,
    order_no      VARCHAR(32)      NOT NULL UNIQUE,   -- PO-YYYYMMDD-NNNN
    created_by    BIGINT           NOT NULL REFERENCES users(id),
    supplier      VARCHAR(100),
    status        purchase_status  NOT NULL DEFAULT 'pending',
    total_amount  NUMERIC(12,2)    NOT NULL DEFAULT 0,
    remark        TEXT,
    created_at    TIMESTAMPTZ      NOT NULL DEFAULT now(),
    paid_at       TIMESTAMPTZ,
    returned_at   TIMESTAMPTZ,
    received_at   TIMESTAMPTZ,
    updated_at    TIMESTAMPTZ      NOT NULL DEFAULT now()
);

CREATE INDEX idx_po_status_time ON purchase_orders (status, created_at DESC);
-- 独立 created_at 索引 (供 v_user_activity 等视图按时间窗过滤)
CREATE INDEX idx_po_created_at  ON purchase_orders (created_at DESC);
```

**状态转换图**:
```
                    ┌─ /pay ──▶  paid ─── /receive ──▶ received  (终态)
   pending ────────┤
                    └─ /return ─▶ returned                       (终态)
```

**双层防御**:
- **Controller 层** (`transitStatus` 用 `updateMany(where:{status:{in: from}})`): 原子地"读+写", 拒绝非法 + 解决 TOCTOU.
- **触发器层** (`fn_purchase_status_change` 兜底 `ELSE RAISE EXCEPTION`): 任何绕过 controller 直连 psql 的 `UPDATE status` 都会被拒绝.

### 2.4 purchase_order_items

```sql
CREATE TABLE purchase_order_items (
    id              BIGSERIAL PRIMARY KEY,
    order_id        BIGINT        NOT NULL REFERENCES purchase_orders(id) ON DELETE CASCADE,
    book_id         BIGINT        REFERENCES books(id),    -- NULL = 新书首次进货
    isbn            VARCHAR(20)   NOT NULL,
    title           VARCHAR(200)  NOT NULL,
    publisher       VARCHAR(100)  NOT NULL,
    author          VARCHAR(100)  NOT NULL,
    purchase_price  NUMERIC(10,2) NOT NULL,
    quantity        INTEGER       NOT NULL CHECK (quantity > 0),
    retail_price    NUMERIC(10,2),
    subtotal        NUMERIC(12,2) GENERATED ALWAYS AS (purchase_price * quantity) STORED
);
```

**重点**:`subtotal` 是 **PostgreSQL 12+ 的 STORED GENERATED COLUMN**,
由 `purchase_price * quantity` 自动计算,**写入时不能赋值**,
保证数据一致性 (永远等于 `price * qty`)。

### 2.6 transactions (财务流水)

```sql
CREATE TYPE transaction_type AS ENUM ('income', 'expense');

CREATE TABLE transactions (
    id            BIGSERIAL PRIMARY KEY,
    type          transaction_type NOT NULL,
    amount        NUMERIC(12,2)    NOT NULL CHECK (amount > 0),
    related_table VARCHAR(32)      CHECK (related_table IN ('purchase_orders','sale_orders')),
    related_id    BIGINT,
    description   VARCHAR(200)     NOT NULL,
    created_by    BIGINT           REFERENCES users(id),
    created_at    TIMESTAMPTZ      NOT NULL DEFAULT now(),
    -- 多态外键 PG 无法强约束, 但至少限制 related_table 取值, 防触发器写入非法值
    CHECK (related_id IS NULL OR related_table IS NOT NULL)
);

CREATE INDEX idx_tx_type_time ON transactions (type, created_at DESC);
CREATE INDEX idx_tx_related   ON transactions (related_table, related_id);
```

### 2.7 operation_logs (审计加分)

```sql
CREATE TABLE operation_logs (
    id           BIGSERIAL PRIMARY KEY,
    user_id      BIGINT        REFERENCES users(id) ON DELETE SET NULL,
    action       VARCHAR(32)   NOT NULL,
    resource     VARCHAR(32)   NOT NULL,
    resource_id  VARCHAR(64),
    ip           VARCHAR(45),
    user_agent   TEXT,

    created_at   TIMESTAMPTZ   NOT NULL DEFAULT now()
);
```

**特色**:
- `payload` 用 JSONB,可索引、可查询、可聚合
- `ON DELETE SET NULL`:删除用户后日志依然保留,user_id 置 NULL
- 中间件异步写入,不阻塞主请求

### 2.8 inventory_alerts (库存预警加分)

```sql
CREATE TABLE inventory_alerts (
    id              BIGSERIAL PRIMARY KEY,
    book_id         BIGINT       NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    threshold       INTEGER      NOT NULL,
    current_stock   INTEGER      NOT NULL,
    last_alerted_at TIMESTAMPTZ  NOT NULL DEFAULT now(),
    resolved        BOOLEAN      NOT NULL DEFAULT FALSE,
    resolved_at     TIMESTAMPTZ
);

-- 部分索引: 只索引未解决的预警
CREATE INDEX idx_alerts_open ON inventory_alerts (resolved, last_alerted_at DESC)
    WHERE resolved = FALSE;
-- 部分唯一索引: 同一本书最多只能有一条未解决预警, 防并发触发器写出双条
CREATE UNIQUE INDEX uq_alerts_book_open
    ON inventory_alerts (book_id) WHERE resolved = FALSE;
```

## 三、触发器与函数

### 3.1 销售扣库存

```sql
CREATE FUNCTION fn_sales_after_insert() RETURNS TRIGGER AS $$
BEGIN
    UPDATE books SET stock = stock - NEW.quantity WHERE id = NEW.book_id;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Book id=% 不存在', NEW.book_id;
    END IF;
    -- 检查库存预警
    PERFORM fn_books_low_stock_check(NEW.book_id);
    RETURN NEW;
END
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sale_items_after_insert
    AFTER INSERT ON sale_order_items
    FOR EACH ROW EXECUTE FUNCTION fn_sales_after_insert();
```

**安全保障**: `books.stock >= 0` 的 CHECK 约束 + 上述触发器 + Prisma 事务,
**三层防护**确保库存不会扣到负数。

### 3.2 销售单写收入流水 (statement-level, SUM 重算)

为了让流水 amount **始终等于 SUM(items.subtotal)** (而非依赖 controller 传入的 total_amount, 杜绝前端篡改 / 浮点 / 重构 bug), 改用 statement-level + NEW TABLE 的触发器, 由 DB 重算:

```sql
CREATE FUNCTION fn_sale_items_after_insert_stmt() RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO transactions (type, amount, related_table, related_id, description, created_by)
    SELECT 'income', SUM(soi.subtotal), 'sale_orders', so.id,
           '销售单 ' || so.order_no, so.created_by
      FROM (SELECT DISTINCT order_id FROM new_items) ni
      JOIN sale_orders      so  ON so.id = ni.order_id
      JOIN sale_order_items soi ON soi.order_id = so.id
     WHERE NOT EXISTS (SELECT 1 FROM transactions tx
                        WHERE tx.related_table='sale_orders' AND tx.related_id=so.id)
     GROUP BY so.id, so.order_no, so.created_by
    HAVING SUM(soi.subtotal) > 0;
    RETURN NULL;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sale_items_after_insert_stmt
    AFTER INSERT ON sale_order_items
    REFERENCING NEW TABLE AS new_items
    FOR EACH STATEMENT EXECUTE FUNCTION fn_sale_items_after_insert_stmt();
```

> [!IMPORTANT] 与 controller 的契约
> Controller 必须用 batch INSERT (`Prisma.join` + `Prisma.sql`) 把同一 sale_order 的所有 items 放进**同一个 statement**, 这样 NEW TABLE 才能含全部行, SUM 才准确。逐条 INSERT 会导致第一个 statement 写入只含第一行的流水, 后续 NOT EXISTS 不再写入 → 错误。

### 3.3 进货四态机 (含非法转移防御 + 入库元数据校验)

```sql
CREATE FUNCTION fn_purchase_status_change() RETURNS TRIGGER AS $$
DECLARE
    item RECORD;
    existing RECORD;
BEGIN
    -- 付款: 写支出流水
    IF OLD.status = 'pending' AND NEW.status = 'paid' THEN
        NEW.paid_at := now();
        INSERT INTO transactions (...) VALUES ('expense', NEW.total_amount, ...);

    -- 退货: 仅记时间
    ELSIF OLD.status = 'pending' AND NEW.status = 'returned' THEN
        NEW.returned_at := now();

    -- 入库: 累加 books.stock, 新书 INSERT, 含元数据一致性校验
    ELSIF OLD.status = 'paid' AND NEW.status = 'received' THEN
        NEW.received_at := now();
        FOR item IN SELECT * FROM purchase_order_items WHERE order_id = NEW.id LOOP
            IF item.book_id IS NULL THEN
                SELECT id, title, author, publisher INTO existing
                  FROM books WHERE isbn = item.isbn;
                IF existing.id IS NOT NULL THEN
                    -- ISBN 已存在: 必须 title/author/publisher 完全一致, 否则视为错配
                    IF existing.title <> item.title
                       OR existing.author <> item.author
                       OR existing.publisher <> item.publisher THEN
                        RAISE EXCEPTION
                          'ISBN % 已存在但元数据不一致: ... vs ...', item.isbn,
                          USING HINT = '请改用已有书的 book_id, 或核对录入信息';
                    END IF;
                    UPDATE books SET stock = stock + item.quantity, ...
                     WHERE id = existing.id;
                ELSE
                    INSERT INTO books (isbn, title, ...) VALUES (item.isbn, ...);
                END IF;
            ELSE
                UPDATE books SET stock = stock + item.quantity WHERE id = item.book_id;
            END IF;
        END LOOP;

    -- 任何其他转移都是非法的, 直接抛出异常拒绝
    ELSE
        RAISE EXCEPTION '非法的进货单状态变更: % -> %', OLD.status, NEW.status
            USING HINT = '合法转移仅有 pending->paid / pending->returned / paid->received';
    END IF;
    RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_purchase_status_change
    BEFORE UPDATE OF status ON purchase_orders
    FOR EACH ROW
    WHEN (OLD.status IS DISTINCT FROM NEW.status)
    EXECUTE FUNCTION fn_purchase_status_change();
```

> [!IMPORTANT] 状态机硬约束的价值
> 即便有人绕过 controller 直接在 psql 跑 `UPDATE purchase_orders SET status='received' WHERE status='pending'`, 触发器也会拒绝, 保证状态机的完整性. 这是"业务规则下沉到数据库"的典型例子.

### 3.4 并发取号 (gen_order_no)

```sql
CREATE FUNCTION gen_order_no(prefix TEXT) RETURNS TEXT AS $$
DECLARE
    today    TEXT   := to_char(now(), 'YYYYMMDD');
    seq      INT;
    lock_key BIGINT := hashtextextended(prefix || today, 0);
BEGIN
    -- 同一天同种订单串行化, 杜绝并发 UNIQUE 冲突
    PERFORM pg_advisory_xact_lock(lock_key);
    SELECT COALESCE(MAX(
        NULLIF(regexp_replace(order_no, '^' || prefix || '-' || today || '-', ''), '')::INT
    ), 0) + 1 INTO seq
      FROM (...) ...;
    RETURN prefix || '-' || today || '-' || lpad(seq::TEXT, 4, '0');
END $$ LANGUAGE plpgsql;
```

**为什么这样设计**:
- `pg_advisory_xact_lock` 是事务级锁, 事务结束自动释放, 不需要显式 unlock
- 锁键用 `hashtextextended(prefix || today)`, 不同日期 / 不同 prefix 之间互不阻塞
- 改用 `MAX(seq) + 1` 而非 `COUNT(*) + 1`, 即便有删除也不会冲突

### 3.5 库存预警

```sql
CREATE FUNCTION fn_books_low_stock_check(p_book_id BIGINT) RETURNS VOID AS $$
DECLARE
    b RECORD;
BEGIN
    SELECT id, stock, low_stock_threshold INTO b FROM books WHERE id = p_book_id;
    IF b.stock <= b.low_stock_threshold THEN
        IF EXISTS (SELECT 1 FROM inventory_alerts WHERE book_id = p_book_id AND NOT resolved) THEN
            UPDATE inventory_alerts SET last_alerted_at = now(), current_stock = b.stock
             WHERE book_id = p_book_id AND NOT resolved;
        ELSE
            INSERT INTO inventory_alerts (book_id, threshold, current_stock)
                 VALUES (p_book_id, b.low_stock_threshold, b.stock);
        END IF;
    ELSE
        UPDATE inventory_alerts SET resolved = TRUE, resolved_at = now()
         WHERE book_id = p_book_id AND NOT resolved;
    END IF;
END $$ LANGUAGE plpgsql;
```

## 五、范式分析

### 5.1 1NF (原子性)

✅ 所有字段都是原子类型 (无数组、无嵌套)。
唯一的 JSONB 字段 `operation_logs.payload` 是审计载荷,**业务上不参与查询**,属于"日志原文",符合 1NF 的语义。

### 5.2 2NF (无部分依赖)

✅ 所有表都使用 BIGSERIAL 代理键作为单字段主键 → 不存在复合主键 → 没有部分依赖。

### 5.3 3NF (无传递依赖)

✅ 仔细检查每张表:
- `users`: 所有字段直接依赖于 id
- `books`: 同上;`retail_price` 与 `low_stock_threshold` 互不依赖
- `purchase_orders`: `total_amount` 由明细汇总而来,**但我有意识地存储**——为审计冻结
- `*_items.subtotal`: STORED GENERATED 列,数学派生,**不算传递依赖**
- `transactions`: 所有字段直接依赖于 id

### 5.4 BCNF

✅ 所有 FD 左侧均为超键。

### 5.5 有意识的反规范化

**`transactions.amount` ↔ `purchase_orders.total_amount`** 存在数值冗余,
这是**事实表 (Event Sourcing) 模式**:
- 订单可改 (例如客户退货后改总金额)
- 流水不可改 (财务监管要求)
- 二者通过 `related_table` + `related_id` 关联,任何时候都可对账

在企业 ERP 系统中,这种设计被称为 "**audit trail**",
不视为违反范式,而是出于业务正确性的有意设计。

## 六、关键索引性能分析

### 6.1 GIN 三元组索引

```sql
-- 默认 B-tree 对 ILIKE '%xx%' 无效
EXPLAIN SELECT * FROM books WHERE title ILIKE '%算法%';
-- → Seq Scan on books (cost=0.00..580 rows=1)

-- 加上 GIN 三元组索引后
CREATE INDEX idx_books_title_trgm ON books USING gin (title gin_trgm_ops);
EXPLAIN SELECT * FROM books WHERE title ILIKE '%算法%';
-- → Bitmap Heap Scan on books (cost=4.30..40)
--    └─ Bitmap Index Scan on idx_books_title_trgm (cost=0.00..4.30)
```

百万级数据上,**GIN 比 Seq Scan 快 100×+**。

### 6.2 部分索引

```sql
CREATE INDEX idx_books_low_stock ON books (stock)
    WHERE stock <= low_stock_threshold;

-- 库存告急查询
EXPLAIN SELECT * FROM books WHERE stock <= low_stock_threshold;
-- → Index Only Scan on idx_books_low_stock
```

只索引"少数低库存书",而非整张表 → 索引体积小,扫描快。

### 6.3 复合索引

```sql
CREATE INDEX idx_po_status_time ON purchase_orders (status, created_at DESC);

-- "未付款进货单按时间倒序"
EXPLAIN SELECT * FROM purchase_orders WHERE status = 'pending' ORDER BY created_at DESC LIMIT 10;
-- → Index Scan on idx_po_status_time (cost=0.28..0.85)
```

## 七、外键与级联策略

| 关系 | 行为 | 原因 |
|------|------|------|
| `purchase_order_items.order_id → purchase_orders.id` | `ON DELETE CASCADE` | 订单删除连带明细 |
| `sale_order_items.order_id → sale_orders.id`         | `ON DELETE CASCADE` | 同上 |
| `purchase_order_items.book_id → books.id`            | (默认 RESTRICT) | 引用书籍不可删除 |
| `sale_order_items.book_id → books.id`                | (默认 RESTRICT) | 同上 |
| `operation_logs.user_id → users.id`                  | `ON DELETE SET NULL` | 用户删了日志要留 |
| `inventory_alerts.book_id → books.id`                | `ON DELETE CASCADE` | 书删了预警自然失效 |
| `transactions.created_by → users.id`                 | (默认 RESTRICT) | 历史财务不允许丢失关联 |
