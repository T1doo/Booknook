## 一、实验背景与需求分析

### 1.1 题目复述

> [!IMPORTANT] PPT 原题
> 某书城需要一套图书管理系统，对图书的进货、销售、财务等方面进行统一管理。系统应支持两类用户（超级管理员 / 普通管理员），覆盖**用户管理、库存维护、多字段查询、信息修改、图书进货、付款、退货、入库、销售、财务管理、查看账单**共 11 项基础功能。系统完成时应已存在一个超级管理员账号。

PPT 还鼓励完成创新性的加分项，**上限为总分的 20%**。

### 1.2 需求拆解到 5 大业务域 + 加分项

我把 11 项需求映射到 5 个业务域，每个域由一个或多个后端模块负责：

| 业务域 | PPT 需求点 | 后端模块 | 前端页面 |
|---|---|---|---|
| **用户与认证** | #1 用户管理 | `auth/`, `users/` | `/login`, `/users`, `/profile` |
| **库存管理** | #2 维护 / #3 查询 / #4 修改 | `books/` | `/books` |
| **进货流程** | #5 进货 / #6 付款 / #7 退货 / #8 入库 | `purchases/` | `/purchases` |
| **销售收银** | #9 书籍购买 | `sales/` | `/sales` |
| **财务管理** | #10 财务 / #11 查看账单 | `transactions/` | `/transactions` |
| **加分（自选）** | 创新性 | `analytics/`, `alerts/`, `logs/`, `reports/` | `/dashboard`, `/alerts`, `/logs`, `/reports` |

**加分项一览**（详见 [docs/创新点说明.md](./创新点说明.md)）：
1. 数据可视化 Dashboard（4 KPI + 4 图表）
2. 库存预警系统（触发器驱动）
3. 操作日志审计（JSONB + before/after 留痕）
4. 报表导出（XLSX + PDF）
5. 国际化（中英双语全覆盖）
6. 暗色模式（暖书店主题）
7. 严格 RBAC（三层防御）
8. 全栈 TypeScript（类型安全）
9. shadcn/ui 设计系统（暖书店主题）
10. helmet + 速率限制 + body limit（Web 安全加固）

### 1.3 与现实书店业务的差异（诚实地说哪些简化了）

为聚焦数据库设计而非业务复杂度，本项目做了以下简化：

- **未实现退款流程**：销售单一旦完成即终态，没有处理"客户买完想退"的路径
- **未实现会员系统 / 积分**：销售单的 `customer_note` 只是文本备注，不关联用户表
- **未实现多分店 / 多仓库**：库存是全局唯一一份，不区分门店
- **未实现库存盘点 / 调拨**：库存只通过销售（减）和入库（加）变动
- **未实现多语言书籍元数据**：只有 UI 层做了中英切换，数据库存的书名是原文

这些简化让数据模型保持在「10 张表 + BCNF」的可论证范围内，符合中期实验的体量。

## 二、系统总览

### 2.1 技术栈选型与理由

| 层 | 技术 | 选型理由 |
|---|---|---|
| 数据库 | **PostgreSQL 16** | 选 PG 而非 MySQL：完整支持课程要求的所有特性——CHECK 约束、ENUM、GENERATED COLUMN、JSONB、部分索引、GIN 三元组、PL/pgSQL 触发器。MySQL 部分特性受限或语法繁琐。 |
| ORM | **Prisma 5** | 选 Prisma 而非裸 SQL：自动生成强类型 Client，编译时捕获字段拼写错误；同时保留 `$executeRaw` 模板字面量出口，触发器交互和复杂聚合不被 ORM 限制。 |
| 后端框架 | **Express + TypeScript** | 选 Express 而非 NestJS：体量适中，中间件链清晰，符合中期实验「能讲清楚」的要求。TypeScript 与 Prisma 完美配合。 |
| 前端框架 | **Next.js 15 + React 19** | 选 Next.js 而非纯 Vite + React：App Router 路由分组（`(app)/*` 包需登录页面）天然支持鉴权布局；SSR 能力虽然没用到，但留有扩展余地。 |
| UI 库 | **shadcn/ui + Radix Primitives + Tailwind** | shadcn 不是依赖而是源码（copy-paste），可定制；Radix 提供无障碍 primitive；Tailwind 让暖书店主题（米色 + 琥珀）落地容易。 |
| 图表 | **Recharts** | React 原生 SVG 渲染，渐变与动画自然。 |
| 文件导出 | **ExcelJS + PDFKit** | ExcelJS 流式输出，不爆内存；PDFKit 中文字体支持完善。 |
| 鉴权 | **JWT (HS256) + HttpOnly Cookie + MD5+salt** | MD5 是 PPT 硬性要求，加 salt 抵御彩虹表；JWT + Cookie 平衡无状态扩展性和 XSS 安全性。 |
| 状态管理 | **Zustand + persist** | 比 Redux 轻量，比 Context 性能好；持久化登录态到 localStorage，刷新页面 UI 不闪烁。 |

**关键设计原则**：
- **三层分离**：浏览器只渲染 UI；Express 处理鉴权、校验、聚合；PostgreSQL 封装业务规则（触发器扣库存 / 写流水 / 状态机）。
- **业务规则下沉到数据库**：销售扣库存、进货状态机、流水产生全部由触发器驱动，确保绕过应用代码（比如 psql 直连）也不能破坏业务一致性。
- **应用层做事务编排 + 输入校验**：Prisma `$transaction` 包裹销售结账，Zod 严校验所有输入边界。

### 3.3 BCNF 范式分析

> [!NOTE] BCNF 定义
> 关系 R 满足 BCNF，当且仅当对每一非平凡函数依赖 `X → Y`，`X` 必为 R 的超键。

逐表论证：

#### `users`
- **候选键**: {id}, {username}, {employee_no}（三者皆 UNIQUE）
- **所有非主属性**（password_hash, salt, real_name, role, gender, age, is_active, ...）都完全依赖于这三个候选键中的任一个
- **不存在传递依赖**: 比如不存在 `username → real_name` 这种依赖（同名不同人是允许的）
- **结论**: 所有非平凡 FD 左侧都是超键 → ✅ BCNF

#### `books`
- **候选键**: {id}, {isbn}
- 非主属性 title, publisher, author, retail_price, stock 等都完全依赖于候选键
- **不存在传递依赖**: 不会出现 `publisher → publisher_address` 这种二级依赖
- **结论**: ✅ BCNF

#### `purchase_orders`
- **候选键**: {id}, {order_no}
- 时间戳 `paid_at` / `returned_at` / `received_at` 直接依赖于 id
- `status` 直接依赖于 id（不传递）
- **结论**: ✅ BCNF

#### `purchase_order_items` / `sale_order_items`
- **候选键**: {id}
- `subtotal` 是 `GENERATED ALWAYS AS (price * quantity) STORED`，物理上存储但**不视为数据冗余**——它是确定性派生值，与公式 `purchase_price × quantity` 严格等价。范式理论中，派生属性（derived attribute）独立于规范化考量。
- **结论**: ✅ BCNF

#### `transactions`
- **候选键**: {id}
- `type` / `amount` / `description` 直接依赖于 id
- `related_table` + `related_id` 是**多态外键**，无法用 PG FK 强约束，但用 CHECK 限制 `related_table` 取值集合
- **结论**: ✅ BCNF

#### 关于"事实表冗余"的辩护

`transactions.amount` 与对应订单的 `total_amount` 数值上重复，这看似违反规范化原则，但属于**审计追溯模式（event sourcing）**：
- 订单可以被修改、状态可以推进
- 但流水一旦写入，**不允许任何 UPDATE/DELETE**
- 这符合企业财务系统设计准则（SOX、COSO 框架）

在数据库教学中，这归类为"**有意识的反规范化（intentional denormalization）**"，是设计决策而非范式违反——前提是有明确的业务理由（审计、不可篡改性）。

### 3.4 约束与触发器

#### 3.4.1 域约束（CHECK）

```sql
-- 库存非负（防超卖最后一道兜底）
ALTER TABLE books ADD CHECK (stock >= 0);

-- 价格非负
ALTER TABLE books             ADD CHECK (retail_price >= 0);
ALTER TABLE purchase_orders   ADD CHECK (total_amount >= 0);
ALTER TABLE transactions      ADD CHECK (amount > 0);

-- 年龄合理性
ALTER TABLE users ADD CHECK (age IS NULL OR age BETWEEN 16 AND 100);

-- 多态外键的部分校验
ALTER TABLE transactions
  ADD CHECK (related_id IS NULL OR related_table IS NOT NULL);
ALTER TABLE transactions
  ADD CHECK (related_table IN ('purchase_orders', 'sale_orders'));
```

#### 3.4.2 引用完整性（FOREIGN KEY）

- `purchase_order_items.order_id → purchase_orders.id` **ON DELETE CASCADE**（订单删除连带明细）
- `sale_order_items.order_id → sale_orders.id` **ON DELETE CASCADE**
- `operation_logs.user_id → users.id` **ON DELETE SET NULL**（审计日志保留用户痕迹，即使用户被删）
- `inventory_alerts.book_id → books.id` **ON DELETE CASCADE**

#### 3.4.3 触发器一览（5 个）

| 触发器 | 类型 | 作用 |
|---|---|---|
| `trg_users_updated_at` / `trg_books_updated_at` / `trg_po_updated_at` | BEFORE UPDATE FOR ROW | 自动维护 `updated_at` 时间戳 |
| `trg_sale_items_after_insert` | AFTER INSERT FOR ROW | 销售明细插入后扣库存 + 触发库存预警检查 |
| `trg_sale_items_after_insert_stmt` | AFTER INSERT FOR STATEMENT | 用 `SUM(items.subtotal)` 写 income 流水 |
| `trg_purchase_status_change` | BEFORE UPDATE OF status FOR ROW | 进货四态机：合法转移写副作用 + 非法转移 RAISE |

#### 3.4.4 触发器深入

##### 销售扣库存（行级触发器）

```sql
CREATE FUNCTION fn_sales_after_insert() RETURNS TRIGGER AS $$
BEGIN
    UPDATE books SET stock = stock - NEW.quantity WHERE id = NEW.book_id;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Book id=% 不存在', NEW.book_id;
    END IF;
    PERFORM fn_books_low_stock_check(NEW.book_id);
    RETURN NEW;
END $$ LANGUAGE plpgsql;
```

**设计要点**：
- 每条 sale_order_items 插入后都触发一次（行级）
- 库存不足时 `CHECK (stock >= 0)` 自动报错并回滚整个事务
- 调用 `fn_books_low_stock_check` 维护预警

##### 销售流水写入（语句级触发器）

```sql
CREATE FUNCTION fn_sale_items_after_insert_stmt() RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO transactions (type, amount, related_table, related_id, description, created_by)
    SELECT
        'income',
        SUM(soi.subtotal),     -- 流水金额由 DB 重算, 不依赖 controller
        'sale_orders',
        so.id,
        '销售单 ' || so.order_no,
        so.created_by
    FROM (SELECT DISTINCT order_id FROM new_items) ni  -- 取本次 statement 全部新行
    JOIN sale_orders      so  ON so.id      = ni.order_id
    JOIN sale_order_items soi ON soi.order_id = so.id
    WHERE NOT EXISTS (
        SELECT 1 FROM transactions tx
        WHERE tx.related_table = 'sale_orders' AND tx.related_id = so.id
    )
    GROUP BY so.id, so.order_no, so.created_by
    HAVING SUM(soi.subtotal) > 0;
    RETURN NULL;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sale_items_after_insert_stmt
    AFTER INSERT ON sale_order_items
    REFERENCING NEW TABLE AS new_items
    FOR EACH STATEMENT EXECUTE FUNCTION fn_sale_items_after_insert_stmt();
```

**设计要点**：
- 用 `REFERENCING NEW TABLE` 取本次 statement 的**全部新行**
- 按 `SUM(items.subtotal)` 重算流水金额——**完全不依赖** controller 传入的 `total_amount`
- 这意味着即使前端篡改 `total_amount`，流水也只反映明细的真实 SUM
- `WHERE NOT EXISTS` 保证幂等：同一销售单不会写两条流水
- **契约**：controller 必须用 batch INSERT（`Prisma.join` + `Prisma.sql`）把同一 sale_order 的所有 items 放进一个 statement，否则 statement-level 触发器拿不到完整数据

##### 进货状态机（最复杂的一个）

```sql
CREATE FUNCTION fn_purchase_status_change() RETURNS TRIGGER AS $$
DECLARE item RECORD; existing RECORD;
BEGIN
    -- 分支 1: 付款 → 写支出流水
    IF OLD.status = 'pending' AND NEW.status = 'paid' THEN
        NEW.paid_at := now();
        INSERT INTO transactions (...) VALUES ('expense', NEW.total_amount, ...);

    -- 分支 2: 退货 → 仅打标记
    ELSIF OLD.status = 'pending' AND NEW.status = 'returned' THEN
        NEW.returned_at := now();

    -- 分支 3: 入库 → 遍历明细, 已有书 +stock, 新书 INSERT
    ELSIF OLD.status = 'paid' AND NEW.status = 'received' THEN
        NEW.received_at := now();
        FOR item IN SELECT * FROM purchase_order_items WHERE order_id = NEW.id LOOP
            IF item.book_id IS NULL THEN
                SELECT * INTO existing FROM books WHERE isbn = item.isbn;
                IF existing.id IS NOT NULL THEN
                    -- 元数据一致性校验: 防"老书改名"事故
                    IF existing.title <> item.title
                       OR existing.author <> item.author
                       OR existing.publisher <> item.publisher THEN
                        RAISE EXCEPTION 'ISBN % 已存在但元数据不一致: ...', item.isbn;
                    END IF;
                    UPDATE books SET stock = stock + item.quantity WHERE id = existing.id;
                ELSE
                    INSERT INTO books (isbn, title, ..., stock) VALUES (...);
                END IF;
            ELSE
                UPDATE books SET stock = stock + item.quantity WHERE id = item.book_id;
            END IF;
        END LOOP;

    -- 关键: 任何不在合法转移内的状态变更都报错, 防绕过 controller
    ELSE
        RAISE EXCEPTION '非法的进货单状态变更: % -> %', OLD.status, NEW.status;
    END IF;
    RETURN NEW;
END $$ LANGUAGE plpgsql;
```

**设计要点**：
- **状态机硬约束**：合法转移只有 3 条（pending→paid / pending→returned / paid→received），其他任何 UPDATE 都会 `RAISE EXCEPTION`
- **元数据一致性**：入库时如果 ISBN 命中已有书，必须 title/author/publisher 完全一致，否则拒绝（防止操作员把老书录入为新书时静默覆盖元数据）
- **触发器条件 `WHEN OLD.status IS DISTINCT FROM NEW.status`**：保证幂等 UPDATE 不会重复执行（操作员不小心点了两次付款，不会双写流水）

##### 订单号生成（并发安全）

```sql
CREATE FUNCTION gen_order_no(prefix TEXT) RETURNS TEXT AS $$
DECLARE
    today    TEXT   := to_char(now(), 'YYYYMMDD');
    seq      INT;
    lock_key BIGINT := hashtextextended(prefix || today, 0);
BEGIN
    PERFORM pg_advisory_xact_lock(lock_key);  -- 同日同 prefix 串行化

    SELECT COALESCE(MAX(
        NULLIF(regexp_replace(order_no, '^PO-' || today || '-', ''), '')::INT
    ), 0) + 1 INTO seq
      FROM purchase_orders WHERE order_no LIKE 'PO-' || today || '-%';

    RETURN prefix || '-' || today || '-' || lpad(seq::TEXT, 4, '0');
END $$ LANGUAGE plpgsql;
```

**设计要点**：
- 原 `COUNT(*) + 1` 在并发下两个事务可读到相同 count → 双写同一订单号 → UNIQUE 冲突
- 用 `pg_advisory_xact_lock` 把"同一天同种 prefix"的取号**串行化**：
  - 锁键: `hashtextextended('PO' || '20260526')`
  - 锁粒度: 事务级，事务结束自动释放
  - 不影响**不同日期 / 不同 prefix** 的并行性能（高峰期 PO 和 SO 不互锁，今天和明天不互锁）
- 用 `MAX(seq) + 1` 代替 `COUNT(*) + 1`：即便历史上有删除，也不会取到已用过的号

##### 库存预警

```sql
CREATE FUNCTION fn_books_low_stock_check(p_book_id BIGINT) RETURNS VOID AS $$
DECLARE b RECORD;
BEGIN
    SELECT id, stock, low_stock_threshold INTO b
      FROM books WHERE id = p_book_id;

    IF b.stock <= b.low_stock_threshold THEN
        -- 已有未解决预警 → 仅更新时间戳; 否则新建
        IF EXISTS (SELECT 1 FROM inventory_alerts
                    WHERE book_id = p_book_id AND resolved = FALSE) THEN
            UPDATE inventory_alerts SET last_alerted_at = now(), current_stock = b.stock
             WHERE book_id = p_book_id AND resolved = FALSE;
        ELSE
            INSERT INTO inventory_alerts (book_id, threshold, current_stock)
            VALUES (p_book_id, b.low_stock_threshold, b.stock);
        END IF;
    ELSE
        -- 库存回升 → 自动 resolved
        UPDATE inventory_alerts SET resolved = TRUE, resolved_at = now()
         WHERE book_id = p_book_id AND resolved = FALSE;
    END IF;
END $$ LANGUAGE plpgsql;
```

**并发安全保证**：部分唯一索引 `CREATE UNIQUE INDEX uq_alerts_book_open ON inventory_alerts (book_id) WHERE resolved = FALSE` 保证同一本书最多只有一条未解决预警。

### 3.5 索引策略

| 索引 | 类型 | 用途与权衡 |
|---|---|---|
| `idx_books_title_trgm` `_author_trgm` `_publisher_trgm` | GIN + pg_trgm | 加速 `ILIKE '%xx%'` 中间模糊查询。B-tree 对前缀模糊（`'xx%'`）有效，对中间模糊无效；GIN 三元组把字符串切成 3-gram 倒排，毫秒级响应。代价是索引体积约为数据 2-3 倍 + 写入稍慢。**对查询频繁、写入少的图书数据划算**。 |
| `idx_po_status_time` (`status, created_at DESC`) | B-tree 复合 | 列表查询 `WHERE status=? ORDER BY created_at DESC LIMIT N` 走索引扫描而非全表 + 排序。前导列是 status（高选择性筛选） + 后置列是排序方向。 |
| `idx_po_created_at` (`created_at DESC`) | B-tree | 单独的时间索引，供 `v_user_activity` 视图按时间窗过滤使用（前一个复合索引前导列是 status，单独按 created_at 范围扫不会走它）。 |
| `idx_tx_type_time` (`type, created_at DESC`) | B-tree 复合 | 财务流水按收入/支出 + 时间筛选 |
| `idx_logs_user` (`user_id, created_at DESC`) | B-tree 复合 | 操作日志按用户 + 时间查询 |
| `uq_alerts_book_open` (`book_id`) WHERE resolved=FALSE | **部分唯一索引** | 同一本书最多一条未解决预警，由 DB 物理层防止并发触发器双写。WHERE 子句让索引体积只覆盖未解决记录（通常只有少数书），维护成本低。 |

**主动删除的索引**（避免冗余）：
- 原 `idx_books_isbn`：`isbn` 字段已有 `UNIQUE` 约束，PostgreSQL 自动建唯一索引，再建普通索引完全冗余
- 原 `idx_books_low_stock` (`WHERE stock <= low_stock_threshold`)：跨列条件不是 IMMUTABLE，索引维护成本高且无法走查询。改用 `v_low_stock_books` 视图实现相同语义

### 3.7 数据库层封装的业务规则（核心设计哲学）

下表对比"业务规则放应用层 vs 放数据库层"：

| 规则 | 放应用层（不好） | 放数据库层（本项目） |
|---|---|---|
| 销售扣库存 | 后端代码 `UPDATE books SET stock = stock - qty`，新接口忘了写就会漏 | 触发器 `fn_sales_after_insert`，明细一插入就扣 |
| 财务流水 | 后端代码 `INSERT INTO transactions`，新接口忘了写就漏账 | 触发器 `fn_purchase_status_change`、`fn_sale_items_after_insert_stmt` 自动写 |
| 状态机 | 后端代码 `if (current==='pending' && next==='paid')`，psql 直连可绕过 | 触发器 `RAISE EXCEPTION` 拦截非法转移 |
| 库存预警 | 后端代码定期扫表，有延迟 | 触发器实时检查 + 部分唯一索引保并发安全 |
| 订单号 | 后端代码 `COUNT(*) + 1`，并发冲突 | DB 函数 `gen_order_no` + advisory lock |

**核心论点**：业务规则放数据库层，**绕过应用代码也不能破坏数据一致性**。这是企业级数据库设计的精髓——应用代码是"用户与数据库交互的方便接口"，不是"数据正确性的唯一守门员"。

### 4.2 关键业务事务示例：销售结账

这是答辩中**最常被追问**的部分，完整实现见 [backend/src/modules/sales/sales.controller.ts:76-148](../backend/src/modules/sales/sales.controller.ts#L76)。

```typescript
router.post('/', validate('body', createSchema), async (req, res) => {
  const body = req.body as z.infer<typeof createSchema>;
  const bookIds = body.items.map((i) => BigInt(i.book_id));

  const so = await prisma.$transaction(async (tx) => {
    // ① 行锁: 锁定本次涉及的所有 book 行
    //    第二个并发结账会阻塞在这里, 等本事务提交后再读到最新 stock
    const locked = await tx.$queryRaw<...>`
      SELECT id, title, stock, retail_price
        FROM books WHERE id IN (${Prisma.join(bookIds)})
        FOR UPDATE`;
    const map = new Map(locked.map((b) => [b.id.toString(), b]));

    // ② 校验库存 + 按"分"为整数累加金额 (避免浮点累积误差)
    let totalCents = 0n;
    for (const it of body.items) {
      const b = map.get(BigInt(it.book_id).toString());
      if (!b) throw Err.badRequest(`书籍 ${it.book_id} 不存在`);
      if (b.stock < it.quantity) {
        throw Err.badRequest(`《${b.title}》库存不足 (当前 ${b.stock}, 需要 ${it.quantity})`);
      }
      const priceCents = BigInt(Math.round(Number(b.retail_price) * 100));
      totalCents += priceCents * BigInt(it.quantity);
    }
    const total = (Number(totalCents) / 100).toFixed(2);

    // ③ 生成订单号 → 插主表
    const [{ order_no }] = await tx.$queryRaw`SELECT gen_order_no('SO') AS order_no`;
    const order = await tx.saleOrder.create({
      data: { order_no, created_by: BigInt(req.user!.uid),
              customer_note: body.customer_note, total_amount: total },
    });

    // ④ BATCH INSERT items: 一次 statement 完成, 配合 statement-level 触发器
    const valuesSql = body.items.map((it) => {
      const b = map.get(BigInt(it.book_id).toString())!;
      return Prisma.sql`(${order.id}, ${BigInt(it.book_id)}, ${it.quantity},
                         ${new Prisma.Decimal(b.retail_price)})`;
    });
    await tx.$executeRaw(Prisma.sql`
      INSERT INTO sale_order_items (order_id, book_id, quantity, unit_price)
      VALUES ${Prisma.join(valuesSql)}
    `);
    // 此时数据库自动执行:
    //   - fn_sales_after_insert (FOR EACH ROW): 扣库存 + 触发预警
    //   - fn_sale_items_after_insert_stmt (FOR EACH STATEMENT): SUM(subtotal) 写 income 流水

    // ⑤ 应用层对账: items SUM 必须 == sale_orders.total_amount, 否则回滚
    const [{ sum }] = await tx.$queryRaw`
      SELECT COALESCE(SUM(subtotal), 0)::TEXT AS sum
        FROM sale_order_items WHERE order_id = ${order.id}`;
    if (sum !== total) {
      throw Err.internal(`销售单对账失败: items SUM=${sum} vs total=${total}`);
    }

    return order;
  });

  // 任一步失败 → 整个事务回滚 → 库存零损耗
});
```

**这一段代码体现了**：
1. **并发安全**: FOR UPDATE 行锁 + 触发器的 UPDATE 一起保证两个并发结账不会双扣
2. **数值精度**: 按"分"为整数累加，避免 0.1 + 0.2 ≠ 0.3 的浮点误差
3. **多层校验**: Zod schema → 库存数量校验 → 触发器 stock CHECK → 应用层对账
4. **关注点分离**: 业务规则在触发器，事务边界在应用层
5. **可观测性**: 异常的 message 直接告诉用户"《X》库存不足"，前端能正则提取书名做高亮

### 4.3 状态机实现：进货单四态

```
                    ┌──────────────────────┐
                    │   pending（未付款）   │ ← 创建后初始态
                    └──┬─────────────┬─────┘
                       │             │
              [pay]    │             │   [return]
                       ▼             ▼
              ┌─────────────┐   ┌───────────────┐
              │ paid（已付款）│   │ returned（已退货）│ ← 终态
              └──┬──────────┘   └───────────────┘
                 │
                 │ [receive]
                 ▼
              ┌────────────────┐
              │ received（已入库）│ ← 终态
              └────────────────┘
```

**合法转移**：只有 3 条
- `pending → paid` （触发写 expense 流水）
- `pending → returned` （仅打标记）
- `paid → received` （遍历明细，新书 INSERT books，已有书 UPDATE stock）

**非法转移防御（多层）**：

| 层级 | 实现 | 文件 |
|---|---|---|
| ① 前端 | UI 只显示当前状态允许的按钮（如 paid 状态不显示「退货」） | [purchases/page.tsx:144-163](../frontend/src/app/(app)/purchases/page.tsx#L144) |
| ② 应用层 | `transitStatus(id, from, to)` 用 `updateMany(where:{status:{in:from}})` 把"读 + 写"合并到一条原子 SQL | [purchases.controller.ts:138-150](../backend/src/modules/purchases/purchases.controller.ts#L138) |
| ③ 数据库层 | 触发器 `fn_purchase_status_change` 在 ELSE 分支 `RAISE EXCEPTION` | [database/01-schema.sql:515-517](../database/01-schema.sql#L515) |

三层防御中任一层都能独立拦截。**即便绕过前端 + 后端，直接 psql 跑 `UPDATE purchase_orders SET status='received' WHERE status='pending'` 也会被触发器拒绝**。

## 五、安全设计

### 5.1 认证：MD5+salt

#### 5.1.1 实现

```typescript
// backend/src/utils/md5.ts
export function genSalt(): string {
  return randomBytes(12).toString('hex');  // 12 字节 = 24 hex
}
export function md5Hash(password: string, salt: string): string {
  return createHash('md5').update(password + salt, 'utf8').digest('hex');
}
export function verifyPassword(password, salt, expectedHash): boolean {
  const actual = md5Hash(password, salt);
  if (actual.length !== expectedHash.length) return false;
  return timingSafeEqual(Buffer.from(actual, 'hex'),
                          Buffer.from(expectedHash, 'hex'));  // 常量时间比较
}
```

#### 5.1.2 设计权衡

| 维度 | 无 salt 的 MD5 | MD5 + salt（本项目） | bcrypt | argon2id |
|---|---|---|---|---|
| 速度 | 极快 | 极快 | 慢（cost=12 约 100ms） | 慢且 memory-hard |
| 抗彩虹表 | ✗ | ✓ | ✓ | ✓ |
| 抗 GPU 暴力 | ✗ | ✗ | ✓ | ✓✓ |
| 抗 ASIC | ✗ | ✗ | ✓ | ✓ (内存难) |

**结论**：本项目用 MD5 + salt 满足 PPT 课程要求，加密强度**远超无 salt 的传统做法**。但**实际生产环境应升级到 argon2id**（OWASP 推荐）。代码封装在 `utils/md5.ts`，后续替换只需改一处。

### 5.2 授权：JWT HS256 + HttpOnly Cookie

```typescript
// utils/jwt.ts
const ALGO: jwt.Algorithm = 'HS256';  // 显式锁定算法
export function signToken(payload) {
  return jwt.sign(payload, env.JWT_SECRET, {
    expiresIn: env.JWT_EXPIRES_IN, algorithm: ALGO,
  });
}
export function verifyToken(token) {
  // 显式 algorithms 数组防 alg=none 攻击
  return jwt.verify(token, env.JWT_SECRET, { algorithms: [ALGO] });
}

// modules/auth/auth.controller.ts
res.cookie(env.COOKIE_NAME, token, {
  httpOnly: true,           // JS 读不到, 防 XSS
  secure: !env.isDev,       // 生产环境仅 HTTPS
  sameSite: 'strict',       // 防 CSRF
  path: '/api',             // 收窄路径
  maxAge: 7 * 24 * 60 * 60 * 1000,
});
```

**关键安全决策**：
- **HttpOnly cookie 是 token 唯一载体**：响应 body 不返回 token，前端不存 localStorage。即使页面有 XSS，JS 也偷不到 token
- **SameSite=strict + path=/api**：CSRF 攻击面最小化
- **算法显式锁定 HS256**：防 `alg=none` token 伪造、防 RS-HS 算法混淆攻击

### 5.6 操作审计 + 深度脱敏

```typescript
// middlewares/audit.ts (节选)
const SENSITIVE_RE = /password|passwd|secret|token|api[_-]?key|credential|authorization/i;

function scrubDeep(value) {
  if (value == null || typeof value !== 'object') return value;
  if (Array.isArray(value)) return value.map(scrubDeep);
  const clone = {};
  for (const k of Object.keys(value)) {
    if (SENSITIVE_RE.test(k)) {
      clone[k] = '***';
    } else {
      clone[k] = scrubDeep(value[k]);
    }
  }
  return clone;
}
```

**设计要点**：
- **深度递归 + 数组支持**：嵌套结构如 `{items: [{password: 'x'}]}` 也能正确脱敏
- **路由信息提前缓存**：在 `next()` 前提取 `parseRoute()` 结果，避免 Express 子路由 mount 后 `req.path` 被污染
- **before/after 留痕**：controller 在 `req.auditBefore` 埋入修改前快照，中间件合并写入 payload（典型如书价修改）
- **异步写入不阻塞**：`prisma.operationLog.create({ ... }).catch(log)`，写日志失败不影响主流程

## 八、遇到的问题与解决

> 真实开发过程中遇到的 5 个棘手问题及解决方案。

### 8.1 销售流水金额与明细总和不一致（1 分钱差）

**问题表现**：测试时偶尔发现 `transactions.amount = 256.40` 但 `SUM(sale_order_items.subtotal) = 256.39`，差 1 分钱。

**根因分析**：原来的实现是 controller 算好 `total_amount` 后传给后端，后端按 `total_amount` 写流水。中间环节用 JS Number 累加价格（`0.1 + 0.2 = 0.30000000000000004`），浮点误差累积导致 1 分钱误差。

**解决方案**：
1. **DB 层**：把流水写入从 `sale_orders` 行触发器改为 `sale_order_items` 语句级触发器，按 `SUM(items.subtotal)` 重算金额。**流水 amount = 明细 SUM 由 DB 强制保证**
2. **应用层**：所有金额累加按"分"为 BigInt 整数处理，最后再除 100 转回字符串
3. **校验层**：事务结束前再查一次 `SUM(items.subtotal)`，不等于 `total_amount` 立即回滚

**复盘思考**：金额场景永远不要信浮点数。**所有金额累加用整数（最小货币单位）+ 最后一刻转回字符串**，是金融系统的硬规矩。

### 8.2 进货状态机可被 psql 直连绕过

**问题表现**：原来的 controller 只检查 `if (current === 'pending' && next === 'paid')`。但如果运维直接连 psql 跑 `UPDATE purchase_orders SET status='received' WHERE id=1`，会跳过付款直接入库，**财务漏一笔支出流水**。

**根因分析**：业务规则只在应用代码层，绕过应用代码就破防。

**解决方案**：把状态机硬约束下沉到触发器：
```sql
ELSE  -- 任何不合法的转移
    RAISE EXCEPTION '非法的进货单状态变更: % -> %', OLD.status, NEW.status
        USING HINT = '合法转移仅有 pending->paid / pending->returned / paid->received';
END IF;
```

**复盘思考**：**业务关键规则必须在数据库层强制**，应用代码只是「方便接口」，不是「数据正确性的唯一守门员」。

### 8.3 并发取号 UNIQUE 冲突

**问题表现**：压测时偶尔报 `duplicate key value violates unique constraint "purchase_orders_order_no_key"`。

**根因分析**：原 `gen_order_no` 用 `COUNT(*) + 1` 算序号。两个并发事务可能同时读到 `COUNT = 5`，都生成 `PO-20260526-0006`，第二个写入触发 UNIQUE 冲突。

**解决方案**：
```sql
PERFORM pg_advisory_xact_lock(hashtextextended(prefix || today, 0));
SELECT COALESCE(MAX(seq), 0) + 1 INTO seq FROM ...;
```
- 用事务级 advisory lock 把"同一天同种 prefix"的取号串行化
- 不影响不同日期 / 不同 prefix 的并发性能
- 改用 `MAX(seq) + 1` 而非 `COUNT(*) + 1`，即使有删除也不冲突

**复盘思考**：序列号生成在并发场景一定要原子化。advisory lock 比 SERIALIZABLE 隔离级别更轻量。

### 8.4 状态推进 TOCTOU 漏洞

**问题表现**：原 `transitStatus` 是「先 findUnique 读状态，再 update 改状态」两步。并发两次付款可能都先读到 pending，都通过校验，都执行 update → **触发器执行两次 → 双写支出流水**。

**根因分析**：经典的 TOCTOU（Time-Of-Check-Time-Of-Use）漏洞。

**解决方案**：用 `updateMany` 把读 + 写合并到一条原子 SQL：
```typescript
const result = await prisma.purchaseOrder.updateMany({
  where: { id, status: { in: from } },
  data:  { status: to },
});
if (result.count === 0) {
  // 区分: 不存在 / 已在目标态(幂等) / 当前状态不允许
}
```
- DB 层加行锁，两个并发只有一个 `count === 1` 成功
- 另一个 `count === 0` → 查实际状态判断是幂等返回还是错误

**复盘思考**：写库的"读-改-写"模式天然不安全。能用 `UPDATE ... WHERE conditions` 一条 SQL 完成的事，**绝不分两步**。

### 8.5 PowerShell GBK 控制台中文乱码

**问题表现**：第一次跑 `00-init-db.ps1` 时，psql NOTICE 输出显示为 `娉ㄦ剰: 琛?xxx 涓嶅瓨鍦?`，启动后端时看到 `馃尶 BookNook Backend ready`。

**根因分析**：
- 中文 Windows 默认 OEM code page 是 936（GBK）
- SQL 文件是 UTF-8，但 psql 客户端 client_encoding 跟随系统区域 = GBK
- psql 按 GBK 解读 UTF-8 字节流，遇到 `进` 字（UTF-8 `0xE8 0xBF 0x98`）就乱
- pino-pretty 输出的 emoji 也被 GBK 控制台错误渲染

**解决方案**：
1. 脚本里设 `$env:PGCLIENTENCODING = 'UTF8'`（告诉 psql "我发给你的是 UTF-8"）
2. SQL 文件开头加 `SET client_min_messages TO warning;`（抑制冗余的 NOTICE 中文）
3. 后端启动日志去掉 emoji（`🌿 → [OK]`）
4. PowerShell 脚本 chcp 65001 + `[Console]::OutputEncoding = UTF8`

**复盘思考**：跨平台/跨区域的文本输出是个隐蔽坑，编码链路必须每一段都对齐 UTF-8。Windows 中文环境对 emoji 和混合编码极其不友好。

## 九、可改进点（诚实自评）

### 9.1 性能层面

- **未做千万级数据实测**：本项目预置 30 本书 + 几十条订单。GIN 三元组索引的优势在百万级数据上才明显，未在更大数据集上压测
- **未引入 materialized view**：6 个 Dashboard 视图每次查询都重算。生产环境如果 Dashboard 访问量大，应改用 `MATERIALIZED VIEW` + 定时 `REFRESH`
- **未做后端缓存**：`/api/auth/me` 等高频调用没加 Redis 缓存。is_active 校验已有 30s LRU 缓存，但仅本地内存

### 9.2 安全层面

- **MD5 仅满足课程要求**：生产环境应升级到 argon2id 或 bcrypt
- **JWT 不支持主动失效**：用户改密码后旧 token 仍有效到自然过期（最长 7 天）。生产环境应引入 Redis 黑名单 + 短期 access token + 长期 refresh token 模型
- **CSRF token 未额外引入**：依赖 SameSite=strict cookie。极端跨域场景（如 iframe 嵌入）可能需要双重提交 cookie

### 9.3 测试层面

- **单元测试覆盖率不足**：项目配置了 vitest，但只跑了 smoke-test，没写细粒度单测
- **缺乏 fuzzing**：触发器、Zod schema 没做随机输入压测
- **性能基准未建立**：没有 baseline 数字，回归发生时无法量化

### 9.4 功能层面

- **未实现退款流程**：销售单一旦完成即终态
- **未实现多分店 / 库存调拨**：库存全局唯一
- **未实现会员系统 / 积分**：customer_note 仅为文本备注
- **未实现 CSV 批量导入**：已规划但因时间未实现
- **PDF 中文字体未打包**：`backend/assets/fonts/NotoSerifSC-Regular.ttf` 缺失时回退默认字体，中文为方框。代码已 graceful 兜底（logger.warn + 走默认），但 PDF 中文显示效果打折

### 9.5 工程层面

- **CI/CD 未配置**：没接 GitHub Actions，每次发版手工跑 smoke-test
- **未配置 Docker**：部署依赖人工装环境，复现成本高
- **日志未接 ELK / 监控未接 Prometheus**：生产化的可观测性缺失

### 收获

1. **数据库层封装业务规则的威力**：之前的项目习惯把业务规则放应用层，遇到并发问题就一层层加锁补救。这次把销售扣库存、状态机、流水产生全交给触发器，**应用代码空前简洁**，绕过应用代码也不能破坏一致性
2. **类型系统是设计工具不只是检查工具**：Prisma 自动生成的强类型 Client 让重构时编译器精确指出影响范围，10 个业务模块共享 10 张表却没出过字段拼写错误
3. **安全是系统性工程**：MD5+salt / JWT 锁算法 / HttpOnly Cookie / RBAC 三层 / 速率限制 / 输入校验 / 输出脱敏，任何一层失守都可能崩盘。开发时要把"如果有人故意攻击我会怎样"作为常驻问题
4. **演示前的细节决定答辩体验**：Dashboard 数字虚高、登录后白屏、控制台乱码……每一个小问题都会让评委分心。做项目和打磨项目是两个完全不同的阶段

### 一句话总结

> 把数据库当成系统的"骨架"（业务规则的强制者），把应用代码当成"肌肉"（编排事务、整理输入），把前端当成"皮肤"（呈现 + 体验）——三者各司其职，整套系统才能既正确、又快、又好看。
