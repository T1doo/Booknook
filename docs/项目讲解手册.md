## 第 1 章 · 你的项目长什么样（架构总览）

### 1.1 一个比喻：BookNook 就像一家有前台、厨房、仓库的餐厅

想象你开了一家书店，店内有：

| 角色 | 现实餐厅 | 对应到 BookNook |
|------|---------|---------|
| **店里的桌椅、菜单、点餐用的 iPad** | 前台/大堂 | **前端（Frontend）** —— 用户在浏览器里看到的页面 |
| **后厨的厨师和服务员** | 厨房 | **后端（Backend）** —— 处理业务逻辑的服务器程序 |
| **冷库 + 食材记账本** | 仓库 | **数据库（Database）** —— 永久存储所有数据的地方 |

**关键事实**：顾客（你的浏览器）**不能直接进入厨房或仓库**。所有"点菜"请求都必须通过服务员（前端 → 后端 → 数据库），再原路返回。这就是所谓 **三层架构**（Three-tier Architecture）。

### 1.2 用一次"销售一本书"来感受这三层

当你在浏览器里点击"结账"按钮的瞬间，发生了下面这一串事件：

```
[浏览器]
  你点了"《算法导论》× 2 本，结账"
  ↓ (网络请求 HTTP POST)
[前端 Next.js]
  把购物车数据封装成 JSON,发到后端
  ↓ (HTTP POST /api/sales)
[后端 Express]
  1. 检查你的登录身份 (JWT)
  2. 校验数据格式 (Zod)
  3. 执行业务逻辑: 开事务、锁库存、写订单、写明细
  ↓ (SQL 语句)
[数据库 PostgreSQL]
  1. 真的把 sale_orders 行插入
  2. 触发器自动执行:
     - 扣减 books.stock
     - 写入 transactions 财务流水
     - 检查并生成 inventory_alerts
  3. 成功后告诉后端"OK"
  ↑ (SQL 返回结果)
[后端 Express]
  把结果整理成 JSON 响应
  ↑ (HTTP 响应)
[前端 Next.js]
  弹出 toast: "销售成功"
  刷新销售单列表
```

**重要**：你看到的 Dashboard 图表、库存表格、销售单 —— 其实是前端"自己画"出来的；前端拿着后端给的 JSON 数据，用代码把它**渲染**成 HTML 给你看。后端只发数据，不发页面长什么样。

## 第 2 章 · 数据库层（PostgreSQL）

### 2.1 PostgreSQL 是什么

简称 **Postgres**，是一款**关系型数据库管理系统**（RDBMS）。和 MySQL、Oracle 一样的角色，但比 MySQL 功能更强：原生支持 JSONB、GENERATED COLUMN、丰富的索引类型、灵活的触发器，还能自定义类型（ENUM）。

> 课程要求用什么数据库都行（PPT 第 8 页"数据库无限制"）。**我们选 Postgres 是因为它能完整展示数据库课的所有知识点：约束、触发器、索引、视图、函数、范式**。

### 2.2 10 张表怎么设计

打开 [database/01-schema.sql](../database/01-schema.sql) 看完整 DDL。简单表清单：

| 表名 | 一句话说明 | PPT 对应需求 |
|------|----------|------------|
| `users` | 系统用户（超管 + 普通管理员） | PPT-1 用户管理 |
| `books` | 当前在售书籍的库存表 | PPT-2/4 库存管理 |
| `purchase_orders` | 进货单主表（四态机） | PPT-5/6/7/8 进货 |
| `purchase_order_items` | 进货单明细（一单多书） | PPT-5/6/7/8 |
| `sale_orders` | 销售单主表 | PPT-9 销售 |
| `sale_order_items` | 销售单明细 | PPT-9 |
| `transactions` | 财务流水（不可修改的账本） | PPT-10/11 财务 |
| `operation_logs` | 操作审计日志 | 加分项 |
| `inventory_alerts` | 库存预警 | 加分项 |
| `i18n_dict` | 中英文翻译字典 | 加分项 |

### 2.3 为什么这样设计 —— 范式与冗余

#### BCNF 范式（这是数据库课的重点考点）

**第三范式（3NF）** 要求："非主属性不依赖于其他非主属性"。
**BCNF 比 3NF 更严**：要求"每个决定因素都是候选键"，即不允许存在任何非候选键的属性能决定其他属性。

举个例子说明我们为什么把"进货明细"独立成 `purchase_order_items` 而不是塞进 `purchase_orders`：

❌ **错误设计**（违反 BCNF）：
```sql
purchase_orders (id, order_no, isbn, title, qty, price, supplier, ...)
```
一个进货单只能装一本书；如果一单进 3 种书就要重复 3 行 `supplier`，造成冗余。

✅ **正确设计**（我们采用的）：
```sql
purchase_orders       (id, order_no, supplier, status, total, ...)
purchase_order_items  (id, order_id, isbn, title, qty, price, ...)
```
主从两表通过 `order_id` 外键关联。`supplier` 只在主表存一份。

#### 故意保留的冗余 —— 历史快照

你可能注意到 `purchase_order_items` 里有 `isbn`、`title`、`publisher`、`author` 这些字段，明明 `books` 表里也有！这是**冗余**吗？

**不是 bug，是设计**。原因：业务上，进货明细必须**冻结当时的事实**。如果某天书的零售价改了，**两年前的进货单不能跟着变**，否则财务对账就乱了。所以明细表存的是"那一刻的快照"。

> 这是答辩时的一个加分点：当评委问"为什么 items 表里也有 title？这不冗余吗？"——你回答："这是有意保留的冗余，目的是**冻结业务事实、防止历史数据被未来的修改污染**。这种冗余是合理的，违反的是 3NF 的字面定义，但不违反 BCNF 的真实精神。"

### 2.4 PostgreSQL 的高级特性，我们用到了哪些

#### ① GENERATED COLUMN（生成列）

```sql
subtotal NUMERIC(12,2) GENERATED ALWAYS AS (purchase_price * quantity) STORED
```
让数据库自动算"小计 = 单价 × 数量"，**应用程序根本不需要计算**。`STORED` 表示算出来的值**真的写到磁盘**（不是每次查询临时算），所以可以建索引。

#### ② ENUM 类型

```sql
CREATE TYPE purchase_status AS ENUM ('pending', 'paid', 'returned', 'received');
```
等价于"这一列只能是这 4 个字符串之一"，比 `VARCHAR + CHECK` 更优雅。

#### ③ CHECK 约束

```sql
stock INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0)
```
库存永远 ≥ 0。**这是防止超卖的最后一道防线**：哪怕应用代码写错了，数据库也会自己拒绝。

#### ④ GIN + pg_trgm 三元组索引（用于模糊搜索）

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_books_title_trgm ON books USING gin (title gin_trgm_ops);
```
普通 B-tree 索引对 `LIKE '前缀%'` 有效、但对 `LIKE '%关键字%'` 无效。`pg_trgm` 把字符串拆成 3 字母片段（trigram），建 GIN 倒排索引后**任意位置的子串匹配都能用上索引**。

> 答辩时如果问"为什么书名搜索这么快？"——回答："建了 trigram GIN 索引，把书名拆成三字母片段后倒排索引，`ILIKE '%关键字%'` 也能 O(log n) 命中。"

#### ⑤ 触发器（Trigger）

触发器是绑在某个表上的**自动执行的 PL/pgSQL 函数**，在 INSERT / UPDATE / DELETE 前后自动跑。我们用了 5 个：

| 触发器 | 时机 | 做什么 |
|--------|------|--------|
| `fn_sales_after_insert` | 插入 sale_order_items 之后 | 扣 books.stock + 检查预警 |
| `fn_sale_order_after_insert` | 插入 sale_orders 之后 | 写一条 income 财务流水 |
| `fn_purchase_status_change` | purchase_orders.status 改变时 | 付款→写支出流水；入库→更新 books.stock |
| `fn_books_low_stock_check` | 由其他触发器调用 | 检查 stock ≤ 阈值则建预警 |
| `trg_*_updated_at` | UPDATE 之前 | 自动维护 updated_at 时间戳 |

**核心思想**：业务规则（"销售必须扣库存"、"付款必须写流水"）写在**数据库层**，而不是应用层。这样无论用什么语言写后端，规则都生效，不可能被绕过。这是答辩亮点。

#### ⑥ 视图（View）

```sql
CREATE VIEW v_daily_sales_trend AS
SELECT DATE(created_at) AS day, COUNT(*) AS orders, SUM(total_amount) AS total
FROM sale_orders GROUP BY DATE(created_at);
```
视图相当于**"取了别名的查询"**。Dashboard 的图表都是查视图，把复杂聚合 SQL 收敛在数据库层，应用代码只 `SELECT * FROM v_...`，简洁。

## 第 3 章 · 后端层（Node.js + Express + Prisma）

### 3.1 Node.js / Express 是什么

- **Node.js**：一个能让你"用 JavaScript 写服务器程序"的运行时（不是浏览器里那个 JS）。Node 处理 HTTP 请求非常快。
- **Express**：跑在 Node.js 上的最经典 Web 框架。它的核心就是一句话：**"收到请求 → 走一串中间件 → 返回响应"**。
- **TypeScript（TS）**：给 JavaScript 加了"类型注解"。比如你声明 `let x: number = 3`，后面再写 `x = 'hi'` 就会报错。**TypeScript 不会跑在浏览器/Node 里**，它编译后还是 JavaScript，但开发时能帮你抓 90% 的低级错误。

### 3.2 Prisma 是什么 —— ORM 入门

**问题**：写服务器时，你要执行很多 SQL。但手动拼字符串容易出错，还可能被 SQL 注入。

**解决方案**：**ORM（Object-Relational Mapping，对象关系映射）**。它让你用对象的方式操作数据库。比如：

```ts
// 不用写 SQL,Prisma 帮你翻译
const book = await prisma.book.findUnique({ where: { isbn: '9787000000001' } });
// 翻译成: SELECT * FROM books WHERE isbn = '9787000000001';
```

Prisma 还能用 `schema.prisma` 文件**反向同步**数据库结构到 TypeScript 类型 —— 这样你写 `book.titel` 时 TS 会报错（应该是 `book.title`）。

> **重要**：本项目并没有用 Prisma 来建表。表是 `database/01-schema.sql` 用纯 SQL 建的；Prisma 只是"读"了表结构，生成对应的 TS 类型。这样既保留了 PostgreSQL 高级特性（GENERATED COLUMN / 触发器），又享受了 Prisma 的类型安全。

### 3.3 一次请求在后端走过的"中间件流水线"

打开 [backend/src/app.ts](../backend/src/app.ts) 看主入口：

```ts
app.use(pinoHttp({...}));        // ① 日志: 把这次请求打到控制台
app.use(cors({...}));            // ② CORS: 允许前端 3000 端口跨域
app.use(express.json());         // ③ JSON 解析: 把 body 字节流转成 JS 对象
app.use(cookieParser());         // ④ Cookie 解析
app.use('/api/auth', authRouter);              // ⑤ 不需登录: 登录/登出
app.use('/api', authRequired, auditLog);       // ⑥ 以下都要登录 + 审计
app.use('/api/books', booksRouter);            // ⑦ 业务路由
// ... 9 个业务模块
app.use(notFoundHandler);        // ⑧ 404 兜底
app.use(errorHandler);           // ⑨ 错误统一处理
```

**中间件**：可以理解为流水线上的工人。每个中间件检查请求，决定要么"放行交给下一个"（`next()`），要么"直接返回响应"（`res.json(...)`）。

#### 来一个真实例子：用户点"销售"按钮

```
HTTP POST /api/sales {body: 购物车数据}
  │
  ├─ pinoHttp        → 打日志: "POST /api/sales 来了"
  ├─ cors            → "前端来源是 localhost:3000? OK 放行"
  ├─ express.json    → 把字节流变成 JS object
  ├─ cookieParser    → 读 cookie 里的 token
  ├─ authRequired    → 验证 JWT, 把 user 信息塞进 req.user
  ├─ auditLog        → 注册一个"响应结束后写审计日志"的钩子
  └─ salesRouter
      └─ POST /  →  validate('body', createSchema)  ← Zod 数据校验
                 →  实际业务函数 (查库、开事务、写订单)
                 →  res.json({code: 0, data: ...})
  ↓
errorHandler          → (如果上面任何一步抛错, 在这里捕获并返回 400/500)
notFoundHandler       → (如果没匹配到路由, 返回 404)
```

每一层都很短、职责单一，这是 Express 的核心哲学。

## 第 4 章 · 前端层（Next.js + React）

### 4.1 React 是什么

**React** 是 Facebook 开源的前端 UI 库。它的核心思想是：

> **页面 = 组件（Component）的树 + 状态（State）。状态变了，React 自动重新渲染相关组件。**

举例：你写一个"计数器"组件：

```tsx
function Counter() {
  const [count, setCount] = useState(0);    // ← 状态
  return (
    <button onClick={() => setCount(count + 1)}>
      点了 {count} 次
    </button>
  );
}
```

- `useState(0)` 声明一个初始值为 0 的状态。
- 点击按钮调 `setCount`，React 自动把按钮文字更新。
- 你**不用**手动操作 DOM（`document.getElementById('btn').innerText = ...`）—— React 做了。

**JSX 语法**：你看到的 `<button onClick={...}>` 不是真的 HTML，是 JS 的"长得像 HTML"的语法糖。编译后变成 `React.createElement('button', {...})`。

### 4.2 Next.js 是什么

Next.js 是**架在 React 之上**的全栈框架。它额外提供了：

- **基于文件系统的路由**：在 `src/app/books/page.tsx` 写一个组件，访问 `/books` 就会渲染它。**不用配置路由表**。
- **服务端渲染（SSR）**：首次访问页面时由服务器先把 HTML 渲染好返回，加载更快、SEO 友好（本项目用的是 CSR + Client Components，但这个能力可用）。
- **构建优化**：自动压缩 / 代码分割 / 静态资源指纹。

### 4.3 一次"点击-发请求-更新"的前端流程

打开 [frontend/src/app/(app)/sales/page.tsx](../frontend/src/app/(app)/sales/page.tsx)（销售页）：

```tsx
// (伪代码简化版)
export default function SalesPage() {
  const [cart, setCart] = useState([]);          // 购物车状态
  const [orders, setOrders] = useState([]);      // 历史订单状态

  // 第一次进页面时加载历史订单
  useEffect(() => {
    api.get('/sales').then(setOrders);
  }, []);

  async function checkout() {
    const detail = await api.post('/sales', { items: cart });   // ← 调后端
    toast.success('销售成功');
    setCart([]);                                                 // 清空购物车
    setOrders([detail, ...orders]);                              // 把新订单加到列表头
  }

  return (
    <div>
      <CartTable cart={cart} onRemove={...} />
      <Button onClick={checkout}>结账</Button>
      <OrdersTable orders={orders} />
    </div>
  );
}
```

关键点：

- **`useState`**：组件内部的"小数据"。状态一变，React 重画这个组件。
- **`useEffect`**：组件挂载后跑一次。常用来"进页面就加载数据"。

#### 背后流程（核心，答辩必考）
```
[后端] POST /sales
   prisma.$transaction(async tx => {
     // 1. 行级锁 (我们修复 bug 后加的, 防止并发超卖)
     SELECT id, title, stock, retail_price
       FROM books
       WHERE id IN (1, 2, 3)
       FOR UPDATE                                ← 锁住这几行,别的事务等

     // 2. 校验库存
     FOR each item: if (stock < quantity) throw '库存不足'

     // 3. 生成订单号 → 插主表 → 插明细
     INSERT INTO sale_orders (...)
     FOR each item:
       INSERT INTO sale_order_items (...)       ← 触发器自动扣库存 + 写流水
   })

[数据库] 一次销售触发了 3 个触发器:
   ① fn_sales_after_insert (每个明细) 
        → UPDATE books SET stock = stock - qty
        → PERFORM fn_books_low_stock_check (库存预警)
   ② fn_sale_order_after_insert (整单)
        → INSERT INTO transactions (income, ...)
   ③ fn_books_low_stock_check (被①调用)
        → IF stock <= threshold THEN 创建/更新 inventory_alerts
```

**关键 SQL 知识点**：
- `SELECT ... FOR UPDATE` 是**行级排他锁**。事务 A 锁住书 #1 后，事务 B 也想锁 #1 会**阻塞等待**，直到 A 提交或回滚。这就杜绝了"两个并发请求都通过库存校验、结果一起扣到负数"的可能。
- `CHECK (stock >= 0)` 是最后一道保险：即使锁失效，数据库也会拒绝负库存。

## 第 7 章 · 关键机制原理深入

### 7.1 登录与身份验证（JWT + Cookie）

#### 流程

```
[浏览器] POST /api/auth/login {username, password}
              ↓
[后端 auth.controller.ts]
   1. SELECT * FROM users WHERE username = ?
   2. verifyPassword(密码, user.salt, user.password_hash)
        ├─ 计算 md5(密码 + salt) → 32 位十六进制
        └─ 用 timingSafeEqual 跟 user.password_hash 比对
              ↑ 用恒定时间比较, 防止时序攻击
   3. signToken({uid, username, role}) → 'eyJhbGc...'
        这是一段 JWT (JSON Web Token):
        ┌─────────────────────────────────────┐
        │ Header.{uid,username,role,exp}.Sig  │
        └─────────────────────────────────────┘
        Sig = HMAC-SHA256(Header + Payload, JWT_SECRET)
        谁不知道 JWT_SECRET 就伪造不了
   4. res.cookie('booknook_token', token, {httpOnly: true})
        把 token 放进 HttpOnly Cookie:
        ├─ httpOnly: JavaScript 读不到, 防 XSS
        ├─ secure: 仅 HTTPS 时发送 (prod 开)
        └─ sameSite=lax: 防 CSRF
              ↓
[浏览器] 收到 Set-Cookie 头, 自动存进本地
        以后每次访问后端都自动带这个 Cookie
              ↓
[后端 middleware/auth.ts] 后续请求
   ① 从 cookie 或 Authorization 头读 token
   ② verifyToken: HMAC 验签, 通过则解出 {uid, username, role}
   ③ 塞到 req.user 上, 业务代码可以直接用
```

#### 为什么用 JWT 而不是传统 Session

- **Session**：服务器维护一张 `sessionId → user` 的表（通常在 Redis 里）。每个请求查表。
- **JWT**：用户信息直接编码进 token，服务器不需要存储 —— 但代价是 token 一旦签发就在过期前一直有效（不易主动作废）。

我们用 JWT 是因为**它无状态、易扩展**，对于课程项目完全够用。

### 7.2 销售扣库存的并发安全（这是答辩必问的）

**问题场景**：商店只剩 1 本《算法导论》。两个顾客同时来，A 和 B 都点了"购买 1 本"。如果没有保护，可能会出现 stock = -1。

**三层保护**：

| 层级 | 机制 | 作用 |
|------|------|------|
| **应用层** | `SELECT ... FOR UPDATE` 行级锁 | 让两个事务**串行**执行：A 锁住 → B 阻塞 → A 提交 → B 重读 stock=0 → 抛"库存不足" |
| **触发器层** | `UPDATE books SET stock = stock - quantity` | 实际扣减 |
| **约束层** | `CHECK (stock >= 0)` | 假如前两层都失效，数据库自己拒绝把 stock 改成负数 |

> **答辩问答**：
> 问："如何防止超卖？"
> 答："三层保护。应用层用 `SELECT FOR UPDATE` 加行级锁串行化；数据库触发器实际扣减；最后表上有 `CHECK (stock >= 0)` 约束。即使前两层因为 bug 失效，数据库也会拒绝把库存改成负数，回滚事务。"

### 7.3 财务流水的自动产生（数据库驱动业务规则）

PPT 要求"图书进货付款时自动写一条支出流水"。如果在后端代码里写：

```ts
// ❌ 应用层
await prisma.transaction.update({...purchaseOrder, status: 'paid'});
await prisma.transaction.create({type: 'expense', ...});   // 必须不忘记
```

这种写法有个致命问题：**如果未来有人加新接口忘了写第二行 INSERT，财务对账就乱了**。

我们的做法：把规则写在数据库触发器里：

```sql
CREATE TRIGGER trg_purchase_status_change
   BEFORE UPDATE OF status ON purchase_orders
   FOR EACH ROW
   WHEN (OLD.status IS DISTINCT FROM NEW.status)   ← 仅状态真变了才执行
   EXECUTE FUNCTION fn_purchase_status_change();
```

后端只发一句 `UPDATE purchase_orders SET status='paid'`，**触发器自动 INSERT 流水**。规则在 DB 层，绕不开。

### 7.4 库存预警的触发器链

```
销售 INSERT sale_order_items
   │
   ▼
触发器 fn_sales_after_insert
   ├── UPDATE books SET stock = stock - qty
   └── PERFORM fn_books_low_stock_check(book_id)
                  │
                  ▼
            读 books.stock 和 books.low_stock_threshold
                  │
        ┌─────────┴──────────┐
        ▼                    ▼
     stock ≤ 阈值          stock > 阈值
        │                    │
        ▼                    ▼
   未解决的预警存在?      UPDATE inventory_alerts
        │                    SET resolved=true
   ┌────┴────┐               WHERE book_id=?
   ▼         ▼               (库存回升自动清警)
 是: 更新   否: INSERT
```

### 7.5 前端怎么"知道"用户已登录

[frontend/src/components/layout/auth-gate.tsx](../frontend/src/components/layout/auth-gate.tsx) 是关键组件：

```tsx

  if (!hydrated) return <Spinner />;       // 还在确认中
  if (!user)     return redirect('/login'); // 未登录 → 跳登录页
  return children;                          // 已登录 → 渲染子组件
}
```

`(app)/layout.tsx` 把所有需登录的页面都包在 `<AuthGate>` 里。`useAuth` 是 Zustand 全局状态，存当前用户信息。

### 7.6 操作日志（审计）的实现

[backend/src/middlewares/audit.ts](../backend/src/middlewares/audit.ts) 是个 Express 中间件：

```ts
export function auditLog(req, res, next) {
  if (!WRITE_METHODS.has(req.method)) return next();    // 只记 POST/PATCH/PUT/DELETE
  res.on('finish', () => {                              // 响应结束后再写, 不阻塞主流程
    if (res.statusCode >= 400) return;                  // 只记成功操作
    prisma.operationLog.create({
      data: {
        user_id: req.user.uid,
        action: 'create/update/...',
        resource: 'books/users/...',
        ip: req.ip,
        user_agent: ...,
        payload: scrubPayload(req.body),                ← 脱敏密码字段
      }
    });
  });
  next();
}
```

注册在 [app.ts:53](../backend/src/app.ts#L53)：
```ts
app.use('/api', authRequired, auditLog);   // 所有 /api/* 路由都自动走这个
```

效果：用户的每次增删改操作都自动留痕，超管能在 `/logs` 页面查看，**业务代码完全不感知**。这是中间件设计的优雅之处。

## 第 8 章 · 常见问题排查（FAQ）

### Q1：后端启动报 `password authentication failed for user "postgres"`

→ [backend/.env](../backend/.env) 里 `DATABASE_URL` 的密码错了。改成你装 PostgreSQL 时设的密码。

### Q2：后端启动报 `Cannot find module ...`

→ 在 `backend/` 目录下重新跑 `npm install`。

### Q3：前端打开页面是空白 / 卡死

→ 检查浏览器开发者工具（F12）→ Console 看报错。常见原因：
- 后端没启动（前端拿不到数据）
- CORS 错误：检查后端 `.env` 的 `CORS_ORIGINS` 是否包含 `http://localhost:3000`

### Q4：登录提示"用户名或密码错误"

→ 数据库是否初始化过（跑过 `00-init-db.ps1`）？默认账号 `super` / `Admin@2026`，注意区分大小写。

### Q5：销售时报"库存不足"，但明明书还有

→ 检查 `books.stock` 列。可能是之前测试时扣完了。运行 `00-init-db.ps1` 可重置。

### Q6：怎么看数据库里实际的数据？

```powershell
psql -U postgres -d booknook
\dt                                  # 列出所有表
SELECT * FROM books LIMIT 5;
SELECT * FROM users;
SELECT * FROM transactions ORDER BY created_at DESC LIMIT 10;
\q                                   # 退出
```

或者装个图形化工具：[pgAdmin](https://www.pgadmin.org/) / [DBeaver](https://dbeaver.io/)。

### Q7：触发器报错，看不到具体哪里出错

打开后端控制台的输出，触发器抛出的 `RAISE EXCEPTION` 会通过 Prisma 翻译成 JS 错误，里面有 `cause` 字段含原始 PG 错误码。

### Q8：怎么改触发器逻辑？

修改 [database/01-schema.sql](../database/01-schema.sql) 里的 `CREATE OR REPLACE FUNCTION fn_xxx`，然后用 psql 单独执行那段：
```powershell
psql -U postgres -d booknook -c "CREATE OR REPLACE FUNCTION fn_sales_after_insert() ..."
```

或者重新跑 `00-init-db.ps1`（注意会清空数据）。

### Q9：我想清空所有数据但保留表结构

```sql
-- 在 psql 里执行
TRUNCATE TABLE
  sale_order_items, sale_orders,
  purchase_order_items, purchase_orders,
  transactions, operation_logs, inventory_alerts
RESTART IDENTITY CASCADE;
```

`books` / `users` 不清空，否则得重新灌种子数据。

### Q10：能在不重启后端的情况下让修改的代码生效吗？

可以。后端用了 `tsx watch`，**保存代码 → 自动重启**。前端用 Next.js dev server，**保存代码 → 自动热刷新页面**。无需手动重启。

### 评委可能问的问题

1. **"这个项目用了什么数据库？为什么？"**
   → "PostgreSQL 16。因为能完整展示数据库课的所有特性：触发器、ENUM、CHECK 约束、GENERATED COLUMN、GIN 索引、JSONB、视图、函数。MySQL 也能做但部分特性受限。"

2. **"你的表设计满足第几范式？"**
   → "全部满足 BCNF。每张表都有代理主键（BIGSERIAL），所有非键属性只依赖主键。明细表里冗余了书名/作者，这是**有意保留**的历史快照冗余，目的是让订单不受未来书籍信息变化影响。"

3. **"销售怎么防止超卖？"**
   → 见 7.2，讲三层保护。

4. **"流水是怎么自动生成的？"**
   → 见 7.3，讲数据库触发器驱动业务规则。

5. **"用了多少索引？怎么决定建在哪？"**
   → "12 个索引。主键 + 唯一约束自动建；状态字段（如 `purchase_orders.status`）建复合索引 (status, created_at DESC) 加速列表查询；模糊查询的 title/author/publisher 建 GIN trigram；低库存预警用部分索引 `WHERE stock <= threshold` 让索引体积只占少数行。"

6. **"操作日志怎么实现的？"**
   → 见 7.6，讲中间件 + JSONB payload + 密码脱敏。

7. **"为什么用 TypeScript 而不是纯 JavaScript？"**
   → "类型安全。Prisma 把数据库结构反映成 TS 类型，调用时编译器能抓字段拼写错误。这降低了维护成本，特别是 10 个业务模块共同访问 10 张表的情况下。"

### 加分点（评委没问也可以主动提）

- "为了在大数据量下保持快速，我用了 pg_trgm GIN 索引、复合索引、部分索引（partial index）。"
- "财务流水表是 append-only 设计，从应用代码层面只 SELECT、不 UPDATE/DELETE，模拟会计准则的不可篡改原则。"
- "库存预警用'触发器链'实现：销售触发器调用库存检查函数，库存回升时自动 resolved 预警。"
- "审计日志用 JSONB 存请求载荷，**敏感字段自动脱敏**（password/token/apiKey 等正则匹配）。"
- "前后端分离架构，后端通过 JWT + HttpOnly Cookie 保证认证安全；CORS 严格白名单。"
- "用 Prisma 的 `$transaction` + `SELECT FOR UPDATE` 实现强一致销售；数据库层 CHECK 约束作为最后保险。"
