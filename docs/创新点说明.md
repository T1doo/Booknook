## 1️⃣ 数据可视化 Dashboard

一屏内呈现 4 个 KPI + 4 个图表,
管理员开机第一眼即知今日状况。

### 实现要点

**数据层** (PostgreSQL 视图):
```sql
CREATE VIEW v_daily_sales_trend AS
SELECT d::DATE AS day,
       COALESCE(s.order_count, 0)   AS order_count,
       COALESCE(s.total_amount, 0)  AS total_amount
FROM generate_series((CURRENT_DATE - INTERVAL '29 day')::DATE, CURRENT_DATE, INTERVAL '1 day') d
LEFT JOIN LATERAL (...) s ON TRUE;
```

`generate_series` 保证即使某天无销售,图表上也有"零点",避免折线断开。

颜色: 全用 CSS 变量 `hsl(var(--chart-1))..hsl(var(--chart-5))`,

## 2️⃣ 库存预警系统

### 设计理念
"事件驱动" — 销售触发器自动维护预警表,不需要轮询。

### 实现

**触发器** `fn_books_low_stock_check(book_id)`:
```sql
IF b.stock <= b.low_stock_threshold THEN
    IF EXISTS (SELECT 1 FROM inventory_alerts WHERE book_id = p_book_id AND NOT resolved) THEN
        UPDATE ... -- 已有未解决,更新时间
    ELSE
        INSERT INTO inventory_alerts ...
    END IF;
ELSE
    UPDATE inventory_alerts SET resolved = TRUE ...  -- 库存回升,自动解除
END IF;
```

在 `fn_sales_after_insert` 中调用:
```sql
PERFORM fn_books_low_stock_check(NEW.book_id);
```

**前端**:
- 顶栏铃铛 (15 秒轮询 alerts 数量)
- 红点 + 脉冲动画 (`animate-ping`)
- `/alerts` 页可查看 / 调阈值 / 标记已处理

**部分索引** 加速:
```sql
CREATE INDEX idx_alerts_open ON inventory_alerts (resolved, last_alerted_at DESC)
    WHERE resolved = FALSE;
```

只索引"未解决"的预警,大幅缩小索引体积。

### 演示
- 进入 `/sales` 销售《失踪的孩子》(库存 3,阈值 3) 1 本
- 库存变 2,顶栏铃铛立刻显示红点
- 点击进入 `/alerts` → 看到这本书
- 调阈值为 0 → 预警自动消失

## 3️⃣ 操作日志审计

### 设计理念
"无侵入审计" — 业务代码不知道审计存在,中间件自动捕获。

### 实现

**中间件** (`backend/src/middlewares/audit.ts`):
```typescript
export function auditLog(req, res, next) {
  if (!WRITE_METHODS.has(req.method)) return next();   // 只审计写操作
  res.on('finish', () => {
    if (res.statusCode >= 400) return;                 // 只审计成功
    prisma.operationLog.create({
      data: {
        user_id:    req.user ? BigInt(req.user.uid) : null,
        action:     parseRoute(req).action,
        resource:   parseRoute(req).resource,
        resource_id:parseRoute(req).resourceId,
        ip:         req.ip,
        user_agent: req.headers['user-agent'],
        payload:    scrubPayload(req.body),            // 密码字段脱敏
      },
    }).catch(...);                                     // 异步,不阻塞主请求
  });
  next();
}
```

**载荷脱敏** (深度递归, 含数组):
```typescript
function scrubDeep(value) {
  if (value == null || typeof value !== 'object') return value;
  if (Array.isArray(value)) return value.map(scrubDeep);
  const clone = {};
  for (const k of Object.keys(value)) {
    if (/password|secret|token|api_key|credential|authorization/i.test(k)) {
      clone[k] = '***';
    } else {
      clone[k] = scrubDeep(value[k]);  // 任意层级递归
    }
  }
  return clone;
}
```

**Before/After 留痕** (v1.1 增强): 关键变更操作 (如 books PATCH) 在 controller 里埋一个 `req.auditBefore` 快照, audit 中间件把它和 `req.body` (after) 一并写入:

```typescript
// books.controller.ts
router.patch('/:id', async (req, res) => {
  const book = await prisma.book.findUnique({ where: { id } });
  (req as any).auditBefore = {
    title: book.title, author: book.author,
    retail_price: book.retail_price.toString(),
    ...
  };
  await prisma.book.update(...);
});

// audit.ts
const payload = req.auditBefore
  ? { before: scrubDeep(req.auditBefore), after: scrubDeep(req.body) }
  : scrubDeep(req.body);
```

**前端** `/logs` 页 (仅超管):
- 按 用户 / 动作 / 资源 / 时间 筛选
- JSONB 载荷展示, **点击行展开**完整 JSON
- before/after 字段一目了然

**修过的 bug** (v1.1): 原代码在 `res.on('finish')` 里调用 `parseRoute(req)`, 但 Express 子路由 mount 后会临时修改 `req.path`, 导致 logs 里 resource 写成了数字 ID. 修复方式: 在 `next()` 前先 `parseRoute()`, 把结果缓存到闭包变量.

### 演示
- super 登录 → 创建书 / 改书价 / 销售 / 退货
- 进入 `/logs` → 看到所有操作完整记录
- **点击改价的那条 → 展开看到 before/after 完整对比**
- 注意修改密码的记录里, `password` 字段是 `"***"`

## 7️⃣ 严格 RBAC

### 实现 (后端)
```typescript
// middlewares/rbac.ts
export function requireRole(...roles) {
  return (req, _, next) => {
    if (!req.user) throw Err.unauthorized();
    if (!roles.includes(req.user.role)) throw Err.forbidden();
    next();
  };
}

// modules/users/users.controller.ts
router.use(authRequired, requireSuperAdmin);  // 路由级守卫
```

### 实现 (前端 - 三层防御 v1.1)
```typescript
// 1. sidebar.tsx — 菜单隐藏 (UX)
{items.map(it => {
  if (it.superOnly && user?.role !== 'super_admin') return null;
  ...
})}

// 2. components/layout/auth-gate.tsx — RoleGuard 组件
export function RoleGuard({ requiredRole, children }) {
  const router = useRouter();
  const user = useAuth((s) => s.user);
  useEffect(() => {
    if (user && user.role !== requiredRole) {
      toast.error('您没有权限访问该页面');
      router.replace('/dashboard');
    }
  }, [user, requiredRole]);
  if (!user || user.role !== requiredRole) return <Spinner />;
  return children;
}

// 3. app/(app)/users/page.tsx — 用 RoleGuard 包裹页面入口
export default function UsersPage() {
  return <RoleGuard requiredRole="super_admin"><UsersPageInner /></RoleGuard>;
}
```

**三层防御**:
1. 前端侧栏隐藏菜单 (UX)
2. **前端 RoleGuard 拦截路由** (防止 admin1 手输 `/users` 整页加载 + 401 toast 反复弹)
3. 后端 `requireSuperAdmin` 拒绝请求 (Security)

- **直接在地址栏输 `/users` → 自动 replace 到 /dashboard + toast 提示** (前端拦截)

## 1️⃣1️⃣ 并发安全设计 (v1.1 重点加分)

> 数据库教学常考点: 单条 SQL 在并发环境的正确性. 本项目把"业务原子性"做到 DB 层.

### 11.1 取号串行化 (`gen_order_no`)

```sql
CREATE FUNCTION gen_order_no(prefix TEXT) RETURNS TEXT AS $$
DECLARE
    today    TEXT   := to_char(now(), 'YYYYMMDD');
    seq      INT;
    lock_key BIGINT := hashtextextended(prefix || today, 0);
BEGIN
    PERFORM pg_advisory_xact_lock(lock_key);     -- 关键!
    SELECT COALESCE(MAX(...), 0) + 1 INTO seq FROM ...;
    RETURN prefix || '-' || today || '-' || lpad(seq::TEXT, 4, '0');
END $$ LANGUAGE plpgsql;
```

**为什么这样设计**:
- 原 `COUNT(*) + 1` 在两个并发事务下会得到相同 seq → UNIQUE 冲突 → 500
- `pg_advisory_xact_lock` 是**事务级锁**, 事务结束自动释放 (无需 unlock)
- 锁键 `hashtextextended(prefix || today)` 让不同日期 / 不同 prefix 并行不阻塞
- 改用 `MAX(seq) + 1` 而非 `COUNT(*) + 1`, 即使有删除也不会冲突

### 11.2 状态机硬约束 (`fn_purchase_status_change`)

```sql
IF OLD.status = 'pending' AND NEW.status = 'paid' THEN ...
ELSIF OLD.status = 'pending' AND NEW.status = 'returned' THEN ...
ELSIF OLD.status = 'paid' AND NEW.status = 'received' THEN ...
ELSE
    RAISE EXCEPTION '非法的进货单状态变更: % -> %', OLD.status, NEW.status;
END IF;
```

任何人 (包括运维 psql 直连) 试图绕过 controller 跑 `UPDATE purchase_orders SET status='received' WHERE status='pending'` 都会被 DB 拒绝.

### 11.3 Controller TOCTOU 修复

```typescript
// 原代码: 读 + 写非原子, 并发两个 pay 都能通过 from.includes(po.status)
const po = await prisma.purchaseOrder.findUnique({...});
if (!from.includes(po.status)) throw ...;
await prisma.purchaseOrder.update({ data: { status: to } });

// 改后: 用 updateMany 把"读+写"合并到一条原子 SQL
const result = await prisma.purchaseOrder.updateMany({
  where: { id, status: { in: from } },
  data:  { status: to },
});
if (result.count === 0) { ... }  // 区分不存在 / 状态不允许
```

### 11.4 流水写入用 Statement-Level 触发器

```sql
-- 旧: AFTER INSERT ON sale_orders, 用 NEW.total_amount (来自 controller, 不可信)
-- 新: AFTER INSERT ON sale_order_items FOR EACH STATEMENT
--    REFERENCING NEW TABLE AS new_items
--    用 SUM(items.subtotal) 重算 → 流水 amount 由 DB 保证 = 明细 SUM
```

### 11.5 inventory_alerts 并发双条防御

```sql
CREATE UNIQUE INDEX uq_alerts_book_open
    ON inventory_alerts (book_id) WHERE resolved = FALSE;
```

部分唯一索引: 即便两个并发触发器都通过 `IF EXISTS` 判断, 第二个 INSERT 也会被 UNIQUE 拒绝.

### 11.6 演示话术

> "我把业务原子性下沉到数据库层. 比如订单号生成用 `pg_advisory_xact_lock` 串行化, 状态机用触发器硬约束, 销售流水用 statement-level 触发器配合 `REFERENCING NEW TABLE` 重算 SUM. 这样无论控制器代码出什么 bug, 财务一致性 / 状态机完整性都不会被破坏."

## 1️⃣2️⃣ 企业级安全加固 (v1.1 重点加分)

| 维度 | 实现 |
|---|---|
| **JWT secret 弱口令防御** | `env.ts` 用 Zod refine 拒绝包含 `please-change` / `do-not-use` / `example` 的 secret, 启动直接 throw |
| **JWT 算法锁定** | `jwt.verify(token, secret, { algorithms: ['HS256'] })` 防 alg=none / RS-HS 混淆 |
| **is_active 实时校验** | auth 中间件每次请求查 DB (30s LRU 缓存), 软删用户的 token 立即近实时失效 |
| **登录速率限制** | `express-rate-limit` 20 次/15min/IP, 防字典攻击 |
| **HTTP 安全头** | `helmet()` 加 CSP/HSTS/X-Frame/X-Content-Type/Referrer-Policy |
| **Cookie 收窄** | `HttpOnly + SameSite=strict + path=/api`, 移除响应 body 中 token (XSS 偷不到) |
| **请求体限制** | `express.json({ limit: '256kb' })` 防 JSON DoS |
| **Prisma 错误码映射** | `P2002→409` / `P2003→400` / `P2025→404` / `SyntaxError→400` |
| **生产环境兜底** | 500 不再返回原始 message (避免泄露内部表名/列名) |
| **进程级兜底** | `unhandledRejection` + `uncaughtException` 进 fatal 日志后受控退出 |
| **最后超管保护** | 禁用最后一个 `super_admin AND is_active` 时拒绝 |

### 演示话术

> "我做了完整的 web 安全加固. 演示登录, 你可以看到响应头里有 helmet 的 X-Frame-Options 等; 浏览器 devtools localStorage 里**没有** booknook_token (它在 HttpOnly cookie 里); 连续输错密码 20 次会被速率限制. 这些都是 v1.1 加固后做的, 比单纯实现功能更体现工程素养."
