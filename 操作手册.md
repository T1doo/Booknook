## §0 完整环境配置（零基础首次部署）

> **目标**: 从一台干净的 Windows 电脑开始，约 90 分钟让 BookNook 在 `http://localhost:3000` 跑起来。
> **前置假设**: Windows 10/11，会用 PowerShell，没装过 Node/PG 也没关系。

### 0.1 系统要求

| 项 | 要求 | 备注 |
|---|---|---|
| 操作系统 | Windows 10/11（也支持 macOS/Linux，但本文以 Windows 为主） | 中文系统 OK |
| 内存 | 8 GB+ 推荐 | 4 GB 也能跑，但前端编译可能慢 |
| 磁盘 | 5 GB 空闲 | 主要被 `node_modules` 占 |
| 网络 | 能访问 npm 源 | 国内可换淘宝镜像（见 §7.3） |

### 0.2 安装 Node.js 20+

Node.js 是前后端的运行环境。

1. 浏览器打开 https://nodejs.org/
2. 点 **LTS 版本**（绿色按钮，标 "Recommended For Most Users"）下载 `.msi`
3. 双击安装，**全部默认点 Next**。安装到 "Tools for Native Modules" 那一步**打勾**（顺手装 build 工具）
4. 安装完毕，**关掉所有 PowerShell 窗口再重开**（让 PATH 生效）

**验证**:
```powershell
node --version    # 期望: v20.x.x 或更高
npm --version     # 期望: 10.x.x 或更高
```

### 0.3 安装 PostgreSQL 16

PostgreSQL 是数据库，所有业务数据都存这里。**这一步最容易踩坑，请仔细操作。**

1. 浏览器打开 https://www.postgresql.org/download/windows/ → 点 "Download the installer"（跳转 EDB 网站）
2. 选 **PostgreSQL 16.x** + **Windows x86-64** 下载（约 350 MB）
3. 双击安装向导：
   - **Installation Directory**: 默认 `C:\Program Files\PostgreSQL\16`
   - **Select Components**: 全选（PostgreSQL Server / pgAdmin 4 / Stack Builder / Command Line Tools）
   - **Data Directory**: 默认
   - **Password**: ⚠️ 这里设置 `postgres` 超级用户的密码，**一定要记住**！比如 `MyDbPwd2026`（**避免用生日/手机号等弱密码**，演示时密码会出现在 `.env` 里）
   - **Port**: 默认 **5432**，**不要改**
   - **Locale**: 默认 `Default locale`
   - 一路 Next 装完（约 5 分钟）
4. 装完弹出的 **Stack Builder**，**直接 Cancel** 跳过

**验证**:
```powershell
psql --version    # 期望: psql (PostgreSQL) 16.x
```

**如果报 "不是内部或外部命令"**：PATH 没加。手动加：
- 右键 "此电脑" → 属性 → 高级系统设置 → 环境变量
- 系统变量找到 `Path` → 编辑 → 新建 → 填入 `C:\Program Files\PostgreSQL\16\bin`
- 确定保存，**重开** PowerShell 再验证

**验证服务是否运行**:
```powershell
Get-Service postgresql*
# 期望: Status = Running
# 如果 Stopped: Start-Service postgresql-x64-16
```

### 0.4 安装 Python 3.10+（可选）

只用于跑 `scripts/smoke-test.py` 端到端测试。不打算跑测试可跳过。

1. 浏览器打开 https://www.python.org/downloads/ → 下载最新版（≥ 3.10）
2. 双击安装，**第一屏底部勾选 "Add python.exe to PATH"**（默认未勾，容易忘）
3. 点 "Install Now"

**验证**:
```powershell
python --version  # 期望: Python 3.10.x 或更高
```

### 0.5 安装 Git

如果你打算 `git clone` 项目代码，需要装 Git。如果你直接下载 zip 包解压，可跳过。

下载地址：https://git-scm.com/download/win，全程默认 Next。

**验证**:
```powershell
git --version     # 期望: git version 2.x.x
```

### 0.6 下载项目代码

**推荐路径**: `E:\Github\Booknook`（路径**不要含中文、空格**）。

**方式 A · git clone**:
```powershell
cd E:\
mkdir Github -ErrorAction SilentlyContinue
cd Github
git clone <项目仓库 URL> Booknook
cd Booknook
```

**方式 B · 下载 zip**:
1. GitHub 项目页 → 绿色 `Code` 按钮 → `Download ZIP`
2. 解压到 `E:\Github\`，把解压出的文件夹重命名为 `Booknook`

**验证目录结构**:
```powershell
cd E:\Github\Booknook
dir
# 期望看到: backend/  frontend/  database/  docs/  scripts/  README.md  操作手册.md
```

### 0.7 配置 backend/.env

后端启动需要一个 `.env` 文件，里面有数据库密码和 JWT 密钥。

**Step 1 · 复制模板**:
```powershell
cd E:\Github\Booknook\backend
Copy-Item .env.example .env
```

**Step 2 · 改数据库密码**:
```powershell
notepad .env
```

找到 `DATABASE_URL=...` 这一行，把 `CHANGE_ME` 替换成你装 PostgreSQL 时设的密码：

```
DATABASE_URL=postgresql://postgres:你的postgres密码@127.0.0.1:5432/booknook?schema=public
```

> 如果密码含特殊字符（`@ # / ?` 等），需要 URL 编码：`@→%40` / `/→%2F` / `#→%23` / `?→%3F`。

**Step 3 · 生成 JWT_SECRET**:

模板里 `JWT_SECRET=<RUN-THE-COMMAND-ABOVE-AND-PASTE-HERE>` 是占位符，后端启动会**拒绝**这个值。生成一个真实密钥：

```powershell
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
```

会输出一长串十六进制字符（96 位），比如：
```
f3e2e69cd224b733d1cf7475fad698aace57098097d5f0a1d445a067526563facd4c41453e638bc2ee4bad8a995b4718
```

复制这串，回到 notepad 里替换 `JWT_SECRET=` 后面的内容，保存关闭。

### 0.8 初始化数据库

项目内有一键脚本，会自动建库 + 灌入种子数据（3 用户 + 30 本书 + 演示订单）。

```powershell
cd E:\Github\Booknook
$env:PGPASSWORD = "你的postgres密码"   # 跟 .env 里同一个
.\scripts\00-init-db.ps1
```

**期望最后看到**:
```
[DONE] BookNook 数据库初始化完毕!
默认账号: super / admin1 / admin2,  密码: Admin@2026
```

**如果 PowerShell 拒绝执行脚本**（"无法加载文件 ... 因为禁止运行脚本"）：
```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
# 输 Y 回车，再重新跑脚本
```

### 0.9 验证数据库连接

```powershell
psql -h 127.0.0.1 -U postgres -d booknook
```

进入交互式终端后，跑几条快查：

```sql
\dt                                     -- 应该列出 10 张表
\dv                                     -- 应该列出 6 个视图
SELECT COUNT(*) FROM books;             -- 应该是 30
SELECT username, role FROM users;       -- 应该是 super / admin1 / admin2
\q                                       -- 退出
```

✅ 看到 30 本书 + 3 个用户 → 数据库就绪。

## §1 启动与停止

> **第一次启动时三个窗口要按顺序打开。日常启动只需要后两个。**

### 1.0 快捷方式速查（一键脚本）

`scripts/` 目录提供 4 个 PowerShell 脚本，把日常操作压成一行命令。**强烈推荐用脚本而非手动 npm**——脚本含错误兜底（npm/prisma 失败立即停止并报红），首次启动也会自动 `npm install` 和 `npx prisma generate`。

| 场景 | 脚本 | 等价于手动 |
|---|---|---|
| 初始化数据库（首次） | `.\scripts\00-init-db.ps1` | 建库 + 跑 schema/seed/views 三个 SQL |
| 重置数据库（含演示数据） | `.\scripts\01-reset-db.ps1` | 同上（00 的语义化别名） |
| **重置为空白态**（仅 3 账号） | `.\scripts\02-reset-empty.ps1` | 上面 + TRUNCATE books / 进货销售 / 流水 |
| 启动后端 | `.\scripts\10-start-backend.ps1` | cd backend + npm install + prisma generate + npm run dev |
| 启动前端 | `.\scripts\11-start-frontend.ps1` | cd frontend + npm install + npm run dev |
| 冒烟测试 | `python scripts\smoke-test.py` | 跑 16 项端到端测试（需后端已启动） |

**新手「五分钟全栈启动」全程用脚本**（三个 PowerShell 窗口顺序执行）：
```powershell
# 窗口 A · 数据库（一次性）
cd E:\Github\Booknook
$env:PGPASSWORD = "你的postgres密码"
.\scripts\00-init-db.ps1            # 看到 [DONE] 即完成, 这个窗口之后可以关

# 窗口 B · 后端
cd E:\Github\Booknook
.\scripts\10-start-backend.ps1      # 一直挂着, 看到 [OK] ... ready 才算 OK

# 窗口 C · 前端
cd E:\Github\Booknook
.\scripts\11-start-frontend.ps1     # 一直挂着, 看到 ✓ Ready 才算 OK
```

浏览器打开 http://localhost:3000 → 完事。后续 §1.1 / §1.2 是「想知道脚本里到底在干啥」的展开版。

### 1.1 启动后端（窗口 B）

**方式 A · 一键脚本（推荐）**:
```powershell
cd E:\Github\Booknook
.\scripts\10-start-backend.ps1
```
脚本会按顺序: 切到 `backend/` → 检测 `node_modules`（缺则 `npm install`） → 检测 `.env`（缺则从 `.env.example` 拷贝并提示改密码后重跑） → `npx prisma generate` → `npm run dev`。任一步失败会立即报红并退出（不会继续走到下一步白白等待）。

**方式 B · 手动执行**:
```powershell
cd E:\Github\Booknook\backend
npm install                      # 仅首次（约 1-2 分钟）
npx prisma generate              # 仅首次或改了 schema 后
npm run dev
```

**期望（两种方式都一样）**:
```
[hh:mm:ss] INFO: [OK] BookNook Backend ready on http://localhost:4000 (development)
```

> ⚠️ 这个窗口要**一直保持开着**。后端在前台运行，关掉就停了。

### 1.2 启动前端（窗口 C）

**开一个新 PowerShell 窗口**。

**方式 A · 一键脚本（推荐）**:
```powershell
cd E:\Github\Booknook
.\scripts\11-start-frontend.ps1
```
脚本会: 切到 `frontend/` → 检测 `node_modules`（缺则 `npm install`） → `npm run dev`。

**方式 B · 手动执行**:
```powershell
cd E:\Github\Booknook\frontend
npm install                      # 仅首次（约 2-3 分钟）
npm run dev
```

**期望（两种方式都一样）**:
```
   ▲ Next.js 15.5.x
   - Local:        http://localhost:3000
 ✓ Ready in 2.3s
```

> ⚠️ 这个窗口也要**一直保持开着**。

### 1.3 健康检查

```powershell
# 任何 PowerShell 窗口
curl.exe http://localhost:4000/health
# 期望: {"ok":true,"time":"2026-..."}
```

浏览器打开 http://localhost:3000，应该看到棕色书架风格的登录页。

### 1.4 优雅停止

- 切到前端窗口 → `Ctrl + C` → 输 `Y` 回车
- 切到后端窗口 → `Ctrl + C` → 输 `Y` 回车
- PostgreSQL 服务是 Windows 后台服务，**不需要停**

### 1.5 完全重置

如果数据被测乱了，想 30 秒回到出厂状态：

```powershell
cd E:\Github\Booknook
$env:PGPASSWORD = "你的postgres密码"
.\scripts\01-reset-db.ps1
```

> 这会**清空所有业务数据**（除了 seed 的 3 用户 + 30 本书 + 1 张演示进货单 + 2 张演示销售单），不会影响代码。

**前后端要不要停**:
- 前端 (Next.js) 不连数据库，不用停
- 后端 (Prisma) 连着数据库，但脚本用 `DROP DATABASE ... WITH (FORCE)` 会自动断开它，**也不用停**
- 重置完成后**后端会自动恢复**（Prisma 连接池下次请求时重连新库），刷新浏览器即可
- 唯一例外: 重置期间发起的 API 请求会失败（DB 一瞬间不存在），等 `[DONE]` 出现后再操作

**两种重置模式可选**:

| 脚本 | 用户 | 书籍 | 进货销售 | 流水 | 适合场景 |
|---|---|---|---|---|---|
| `01-reset-db.ps1` | 3 | **30 本预置** | 1+2 张演示单 | 3 条 | 给老师/评委展示，Dashboard 有数据更好看 |
| `02-reset-empty.ps1` | 3 | **0**（空） | 0 | 0 | 想演示「从零搭建」完整流程，或展示系统刚部署上线的状态 |

两个脚本都保留 3 个登录账号（PPT 要求「系统完成时便已存在超管」）+ i18n 词条（前端中英切换需要）。区别只在「业务数据是否预置」。

用法：
```powershell
$env:PGPASSWORD = "你的postgres密码"
.\scripts\02-reset-empty.ps1      # 空白态
# 或
.\scripts\01-reset-db.ps1         # 完整 seed 态
```

## §3 PPT 11 项基础功能详细操作

> 每节包含: **操作步骤** + **预期结果** + **背后发生了什么**（DB / 代码层面）。

### 3.1 用户管理（PPT 1）

#### 3.1.1 修改自己的资料（任何角色）

1. 顶栏右上角头像 → `[个人信息]` → 进入 `/profile`
2. 修改真实姓名 / 性别 / 年龄 → `[保存]`
3. 修改密码: 输旧密码 + 新密码（≥ 6 位） → `[修改密码]`

**背后**: `PATCH /api/auth/me`，密码改动用 MD5+salt 重新计算（同一个 salt 保留）。

#### 3.1.2 超管创建新用户

1. 用 `super` 登录 → 侧栏 `[用户管理]` → 进入 `/users`
2. 右上 `[新建用户]` → 弹对话框
3. 填写: 用户名 / 密码 / 真实姓名 / 工号 / 性别 / 角色
4. `[保存]` → 列表立即出现新用户

**背后**: `POST /api/users`，路由前置 `requireSuperAdmin` 中间件，普通管理员调用会 403。

#### 3.1.3 停用/启用用户

- 列表右侧 `[停用]` / `[启用]` 按钮直接切换 `is_active`
- **保护**: 不能停用自己；不能停用最后一个超管

**背后**: 软删除（设 `is_active=false`），保留外键引用完整性；用户的 JWT 在 30 秒内自动失效（auth 中间件实时校验 + 短缓存）。

### 3.2 库存图书 · 查询（PPT 2、3）

![库存图书](./docs/images/03-books.png)

进入 `/books`，看到 30 本预置图书的列表。

**多字段查询**:
- 左上下拉框: `全部 / 编号 / ISBN / 书名 / 作者 / 出版社`
- 右侧搜索框: 输关键词回车

**演示样例**:
| 字段 | 关键词 | 期望结果 |
|---|---|---|
| 作者 | `尤瓦尔` | 命中《人类简史》《未来简史》 |
| 书名 | `算法` | 命中《算法导论》 |
| ISBN | `9787020002207` | 精确命中《红楼梦》 |
| 全部 | `1` | 命中 id=1 的《红楼梦》（数字也按 ID 匹配） |

**库存徽章颜色**: 充足→绿 / 告急→琥珀 / 缺货→红，阈值由 `low_stock_threshold` 决定。

**背后**: 模糊查询用 PostgreSQL **GIN 三元组索引**（`pg_trgm` 扩展），`ILIKE '%xx%'` 在百万级数据上仍是毫秒级。详见 [database/01-schema.sql:136-138](database/01-schema.sql#L136)。

### 3.3 库存图书 · 信息修改（PPT 4）

1. 库存列表 → 点某行的 `[✏️]` 编辑图标 → 弹对话框
2. 可改字段: 书名 / 作者 / 出版社 / 零售价 / 分类 / 库存预警阈值
3. `[保存]` → toast 提示成功 → 列表立即刷新

**背后**: `PATCH /api/books/:id`，audit 中间件自动记录 `before/after` 快照到 `operation_logs` 表，超管可在 `/logs` 查看。

### 3.4 库存图书 · 删除

1. 列表右侧 `[🗑️]` 删除图标
2. **保护**: 库存 > 0 时拒绝；已有销售记录时拒绝
3. 删除某本库存=0 且无销售记录的书 → toast "已删除"

### 3.5 进货管理 · 创建进货单（PPT 5）

![进货管理](./docs/images/04-purchases.png)

进入 `/purchases`，点右上 `[+ 新建进货单]`。

**填写示例**:
- 供应商: `测试出版社`
- 第一行（已有书自动填充）:
  - ISBN: `9787020002207` → **失焦自动填充**《红楼梦》信息
  - 进货价: `30`，数量: `5`
- `[+ 增加一行]`，第二行（新书首次进货）:
  - ISBN: `9787999000001`
  - 书名: `《新书测试》`，出版社: `测试社`，作者: `测试者`
  - 进货价: `25`，数量: `10`
- 右下角合计自动算出 `¥400.00`
- `[提交进货单]` → 列表出现新单，状态 `未付款`（橙色徽章）

**背后**:
- 订单号由 PostgreSQL 函数 `gen_order_no('PO')` 生成，格式 `PO-YYYYMMDD-XXXX`，并发安全（`pg_advisory_xact_lock` 同日同前缀串行化）
- 明细中 `book_id` 对已有书直接引用，对新书暂为 `NULL`，**入库时再回填**
- `subtotal` 是 `GENERATED ALWAYS AS (purchase_price * quantity) STORED` 列，由 DB 计算

### 3.6 进货管理 · 付款（PPT 6）

未付款的进货单，右侧点 `[付款]` → 弹 confirm → 确定。

**预期**:
- 状态 `未付款` → `已付款`（灰色徽章）
- 切到 `/transactions` → 顶部多一条 **支出 ¥400.00 进货单付款** 流水

**背后**: 触发器 `fn_purchase_status_change` 在 `pending → paid` 时自动 INSERT 到 `transactions` 表。后端代码只发一句 `UPDATE purchase_orders SET status='paid'`，流水的产生**不依赖应用层代码**——这是「数据库层封装业务规则」的核心价值。

### 3.7 进货管理 · 退货（PPT 7）

未付款的进货单右侧点 `[退货]` → 状态变 `已退货`。

> **限制**: 已付款的单子无法退货（按钮不显示）。状态机只允许 `pending → returned`，触发器对其他转移会 `RAISE EXCEPTION`。

### 3.8 进货管理 · 入库（PPT 8）

已付款的进货单右侧点 `[入库]` → **弹出入库对话框**（不是简单 confirm）。

**对话框内容**：明细自动按 ISBN 是否已在 books 表里分两类：

| 分组 | UI 表现 | 操作员动作 |
|---|---|---|
| 🆕 **新书**（book_id 为 NULL） | 零售价输入框，红色 `*` 必填，默认预填「进货价 × 1.5」 | 可调整为实际零售价，不填则按钮 disabled |
| 📚 **老书**（已在 books 表） | 只展示「现有零售价」是只读文本 | 不可改（要改请到 `/books` 编辑） |

填完新书价后点 `[确认入库]`：

**预期**:
- 状态 `已付款` → `已入库`（绿色徽章）
- 切到 `/books`:
  - 已有书的库存 +5（红楼梦库存从 48 涨到 53），零售价**不变**
  - 新书自动出现在库存列表，**零售价 = 你刚才填的值**

**背后**: 触发器 `fn_purchase_status_change` 在 `paid → received` 时遍历每条明细，分两种情况:
- 已有书 → `UPDATE books SET stock = stock + qty, retail_price = COALESCE(item.retail_price, retail_price)`（COALESCE 保证「不传就保留原值」）
- 新书 → `INSERT INTO books (..., retail_price)`，并回填 `purchase_order_items.book_id`
- ISBN 一致性校验: 如果新进货的 ISBN 命中已有书但书名/作者/出版社不一致，会 `RAISE EXCEPTION` 拒绝（防"老书改名"事故）

**设计理由**（值得讲给评委）：
- PPT 原话「入库时需要添加书籍的零售价」→ 新书强制必填，符合
- 但书的定价是物理属性（封底印好），同一本书再次进货不应该改价 → 老书默认沿用，业务现实合理
- UI 层的分组和触发器层的 `COALESCE` 一致：触发器留口子但 UI 默认不让改老书，**两层契约一致**

### 3.9 销售收银（PPT 9）

![销售收银](./docs/images/05-sales.png)

进入 `/sales`。

**操作步骤**:
1. 左上搜索框输 `算法` 回车 → 出现《算法导论》
2. 点该书右侧 `[+ 加入]` → 进入右侧购物车
3. 再搜 `百年` → 加入《百年孤独》
4. 右侧购物车 2 项，可 +/− 调数量，可填客户备注
5. 点底部 `[结账]` → toast "结账成功 SO-..."

**预期**:
- 左下「最近销售单」立即出现新单
- 切到 `/books`: 两本书库存各减 1
- 切到 `/transactions`: 多一条 **收入** 流水（金额 = 单价 × 数量之和）

**背后** (这是答辩重点):
1. **行锁**: `SELECT ... FOR UPDATE` 锁住涉及的所有 `books` 行，两个并发结账请求会串行化（不会双方都通过库存校验后双扣到负数）
2. **事务**: `prisma.$transaction` 包裹整个流程
3. **触发器扣库存**: 每条 `sale_order_items` 插入后触发 `fn_sales_after_insert`，`UPDATE books SET stock = stock - qty`
4. **触发器写流水** (statement-level): 一次性按 `SUM(items.subtotal)` 写一条 income 流水，金额由 DB 强制等于明细 SUM（前端篡改无效）
5. **CHECK 约束兜底**: `CHECK (stock >= 0)`，假如前面层都失效，库存改成负数会直接报错并回滚
6. **应用层对账**: 事务结束前再查一次 `SUM(items.subtotal)`，不等于 `total_amount` 立即回滚

详见 [backend/src/modules/sales/sales.controller.ts:76-148](backend/src/modules/sales/sales.controller.ts#L76)。

## §6 答辩 FAQ 预演

> 整理了答辩中**最常被问到的 15 个问题**，含简洁口语化的回答和扩展知识点。

### 6.1 数据库设计类

#### Q1: 你的表设计满足第几范式？

> 全部满足 **BCNF**。每张表都有 BIGSERIAL 代理主键，保证 1NF（原子）和 2NF（无部分依赖）；非主属性也不传递依赖（比如 `subtotal` 是 `GENERATED ALWAYS AS STORED`，由 `quantity × unit_price` 计算，不冗余存储）；所有函数依赖左侧都是超键，所以达到 BCNF。
>
> 唯一"看似冗余"的是 `transactions.amount` 跟订单的 `total_amount` 双写——这是**事实表模式**，订单可改、流水不可改，符合企业财务审计准则。

#### Q2: 用了多少触发器？分别干啥？

> 5 个触发器:
> - `trg_users_updated_at` / `trg_books_updated_at` / `trg_po_updated_at`: 维护 `updated_at` 时间戳
> - `trg_sale_items_after_insert` (FOR EACH ROW): 销售明细插入后扣库存 + 触发预警检查
> - `trg_sale_items_after_insert_stmt` (FOR EACH STATEMENT): 用 `SUM(items.subtotal)` 写 income 流水，金额由 DB 强制等于明细 SUM
> - `trg_purchase_status_change`: 进货状态机，付款写支出流水 / 入库更新库存 / **非法转移直接 RAISE EXCEPTION**

#### Q3: 为什么用 GIN 三元组索引而不是 B-tree？

> 需求是 "模糊搜索书名/作者"，对应 SQL `WHERE title ILIKE '%xx%'`。B-tree 对前缀模糊（`'xx%'`）有效，**对中间模糊无效**；GIN + pg_trgm 把字符串切成三元组（trigrams），即使 `%中%` 也是毫秒级。
>
> **索引选型**是数据库课的核心能力——不同查询模式选不同索引，不是建了就好。

#### Q4: 视图和 ad-hoc query 怎么选？

> Dashboard 用预定义视图（`v_daily_sales_trend` 等），原因:
> - **可维护性**: 聚合逻辑写在数据库一次，前端调多次
> - **可优化**: DBA 看到 view 慢可以升级为 materialized view 不改业务代码
> - **可重用**: 报表导出和 Dashboard 共用同一份聚合逻辑

### 6.2 安全类

#### Q5: 密码为什么用 MD5？

> PPT 明确要求 MD5。但我加了 **12 字节随机 salt**，每个用户独立 salt，即使两人密码相同哈希也不同，**有效抵御彩虹表**。比对密码时用 `timingSafeEqual` 常量时间比较，防时序攻击。
>
> 同时**报告里专门讨论了 MD5 的安全权衡**——生产环境应升级到 bcrypt/argon2 这种慢哈希，让暴力破解成本指数级上升。这一段体现了对密码学的理解深度。

#### Q6: 登录态怎么存？为什么不放 localStorage？

> 用 **JWT (HS256 强锁) + HttpOnly Cookie** 的组合:
> - HttpOnly 让 JS 读不到 token，**XSS 偷不到**
> - SameSite=strict + path=/api 进一步收窄 CSRF 攻击面
> - 算法显式锁 HS256，防 `alg=none` / RS-HS 混淆攻击
>
> 不放 localStorage 是因为只要页面有 XSS，localStorage 里的 token 立即泄露。HttpOnly Cookie 是 OWASP 推荐的浏览器存储方式。

#### Q7: 怎么实现的权限隔离？

> **三层防御**:
> - 后端 `requireSuperAdmin` 中间件挂在 `/api/users` 和 `/api/logs` 前，普通管理员调用返回 403
> - 前端 `RoleGuard` 组件 wrap 页面，admin 进入自动 replace 回 dashboard
> - 侧栏菜单根据 `useAuth().user.role` 条件渲染
>
> 即使前端两层失效，后端兜底保证数据安全。

#### Q8: SQL 注入怎么防？

> Prisma 全部参数化查询，**编译时类型检查**。少数 raw SQL 用模板字面量 `prisma.$executeRaw\`INSERT ... ${val}\``，Prisma 会自动参数化，**不是字符串拼接**。所有用户输入先经 Zod 严校验（类型、长度、枚举范围）。

### 6.3 并发与事务类

#### Q9: 销售怎么防止超卖？

> **三层保护**:
> 1. **应用层**: `SELECT ... FOR UPDATE` 行级锁，让两个并发结账串行化，A 锁住 → B 阻塞 → A 提交 → B 重读 stock=0 → 抛"库存不足"
> 2. **触发器层**: `UPDATE books SET stock = stock - quantity` 实际扣减
> 3. **约束层**: `CHECK (stock >= 0)`，假如前两层失效，数据库自己拒绝把 stock 改成负数

#### Q10: 并发付款会不会被双写流水？

> 不会。`transitStatus` 用 `prisma.purchaseOrder.updateMany({ where: { id, status: { in: ['pending'] } }, data: { status: 'paid' } })`，把**读 + 写合并到一条原子 SQL**。
>
> 数据库层加行锁。两个并发 pay 只有一个会成功（`count === 1`），另一个 `count === 0` 触发幂等返回。这样**触发器最多触发一次**，流水写一次。

#### Q11: 如果有人在 psql 直接 UPDATE 状态会怎样？

> 触发器 `fn_purchase_status_change` 在三个合法分支（`pending→paid` / `pending→returned` / `paid→received`）之外的任何转移都 `RAISE EXCEPTION '非法的进货单状态变更'`。即便直接 `UPDATE purchase_orders SET status='received' WHERE status='pending'` 也会被拒绝。**业务规则在数据库层强制**，应用代码外不可绕过。

#### Q12: 流水金额怎么保证和明细一致？

> 用 **statement-level 触发器** + `REFERENCING NEW TABLE`:
> ```sql
> CREATE TRIGGER trg_sale_items_after_insert_stmt
>     AFTER INSERT ON sale_order_items
>     REFERENCING NEW TABLE AS new_items
>     FOR EACH STATEMENT EXECUTE FUNCTION fn_sale_items_after_insert_stmt();
> ```
> 触发器从 `new_items` 取本次 statement 全部新行，按 `SUM(subtotal)` 写流水。**金额 = 明细 SUM 由 DB 强制**，不依赖 controller 传入的 `total_amount`（防前端篡改和浮点误差）。
>
> 同时事务结束前应用层还会再查一次对账，不一致立即回滚。

### 6.4 工程类

#### Q13: 为什么用 Prisma + raw SQL 混合？

> Prisma 提供类型安全的查询，**90% 场景下高效**；但 `subtotal` 是 `GENERATED ALWAYS AS STORED` 列，Prisma 不支持写入；触发器调用、订单号生成等也需要 raw SQL。所以 `purchase_order_items` / `sale_order_items` 的 INSERT 用 `prisma.$executeRaw\`...\`` ——**取长补短**。

#### Q14: 加分项里你最满意哪个？

> **操作日志审计中间件**——一个 30 行的 `audit.ts` 全局拦截所有写请求，自动记录 user / action / resource / before-after / JSONB payload，密码字段深度递归脱敏。**业务代码完全不感知审计的存在**，符合企业级 RBAC + 审计追溯要求。
>
> 视觉上最满意 **暖书店主题**: 配色（米色 + 琥珀）+ 字体（Playfair Display + Noto Serif SC）+ Logo（CSS 绘制书架）+ Sidebar 深棕调，**每个细节都呼应"图书"的主题**，不是套用通用模板。

#### Q15: 入库时新书的零售价怎么处理？老书呢？

> PPT 要求「入库时录入零售价」，所以**新书首次入库必须填零售价**——前端弹对话框，零售价框为空时「确认入库」按钮 disabled，强制录入。
>
> 老书再次进货时**默认沿用 books 表里的现有零售价**——因为书的定价是物理属性（封底印好），同一本书不该因为再进一批就改价。要调价请去「库存图书」页面单独编辑，避免和「入库」流程耦合。
>
> 触发器层用 `COALESCE(item.retail_price, books.retail_price)` 实现「传了就用新值，不传就保留原值」，UI 层默认不让改老书价格，**两层契约一致**。

#### Q16: 你怎么测试这个系统？

> 三个层级:
> 1. **端到端冒烟测试** `scripts/smoke-test.py`，16 项覆盖: 登录 / Dashboard / 搜索 / 进货全流程 / 销售 / 流水 / 操作日志 / RBAC / 退货 / 库存预警 / 速率限制
> 2. **数据一致性自检**: 跑 SQL 校验 `sale_orders.total = SUM(items.subtotal)` / `transactions.amount = sale_orders.total`
> 3. **手动 demo**: §5 的 10 分钟完整 demo 流程

## §7 常见问题排查

### 7.1 端口被占用

**症状**: 后端启动报 `EADDRINUSE: address already in use 0.0.0.0:4000` / 前端报类似 3000 错误。

**解决**:
```powershell
# 找出占用进程
netstat -ano | findstr :4000
# 看最后一列 PID, 然后:
taskkill /PID <PID> /F
```

或改端口: 后端改 `.env` 的 `PORT=4001`，同时改 `frontend/next.config.mjs` 的代理目标。

### 7.2 psql 密码错 / 连接失败

**症状**: 跑 `00-init-db.ps1` 报 `[ERROR] DROP DATABASE 失败` 或 `password authentication failed`。

**排查**:
```powershell
# 1. PG 服务是否在跑
Get-Service postgresql*
# 如果 Stopped: Start-Service postgresql-x64-16

# 2. 密码是否正确
$env:PGPASSWORD = "你的真实密码"
psql -h 127.0.0.1 -U postgres -d postgres -c "SELECT 1"

# 3. 如果忘了密码, 用 pgAdmin (装 PG 时一起装的) 改密码
```

### 7.3 npm install 慢或卡住

国内访问 npm 官方源经常超时。换淘宝镜像:
```powershell
npm config set registry https://registry.npmmirror.com
# 然后重新 npm install
```

如果之前已经装了一半失败，先清空再装:
```powershell
Remove-Item -Recurse -Force node_modules, package-lock.json
npm install
```

### 7.4 Prisma Client 未生成

**症状**: 后端启动报 `Cannot find module '@prisma/client'` 或 `Property 'xxx' does not exist on type 'PrismaClient'`。

**解决**:
```powershell
cd E:\Github\Booknook\backend
npx prisma generate
```

如果改过 `prisma/schema.prisma`，每次都要重跑。

### 7.5 PowerShell 中文乱码

**症状**: 跑脚本时看到 `馃尶` / `娉ㄦ剰` 这种乱码。

**根因**: PowerShell 5.1 在中文 Windows 上默认 OEM code page 是 936 (GBK)，UTF-8 输出会被错误解码。

**解决**:
- 脚本里已经 `chcp 65001` 切到 UTF-8。如果还乱，**重开 PowerShell** 再跑。
- 用 Windows Terminal 替代经典 PowerShell（默认 UTF-8 友好）。
- 看到 `WARNING: Console code page (65001) differs from Windows code page (936)` 是 psql 的善意提醒，**不影响功能**。

### 7.6 Dashboard 数据为空

**症状**: 登录后 Dashboard 4 张 KPI 卡都是 0。

**根因**: 数据库没 seed 或被重置后 seed 失败。

**解决**:
```powershell
$env:PGPASSWORD = "你的密码"
.\scripts\00-init-db.ps1
# 看最后输出 "用户数 = 3 / 书籍数 = 30 / 进货单 = 1 / 销售单 = 2 / 流水数 = 3" 才算成功
```

如果 seed 后 Dashboard 还是空，浏览器 F12 看 console 是否有 401（cookie 过期，重新登录）。

### 7.7 浏览器白屏

**症状**: 打开 http://localhost:3000 是白屏。

**排查**:
1. F12 → Console 看红色错误信息
2. F12 → Network 看哪个请求失败
3. 后端窗口是否在 ready 状态（看到 `[OK] BookNook Backend ready`）
4. 后端是否真在听 4000 端口: `curl.exe http://localhost:4000/health`

常见原因:
- 后端没启动 → 启动后端
- CORS 错误 → 检查 `backend/.env` 的 `CORS_ORIGINS` 包含 `http://localhost:3000`

### 7.8 登录后立刻被踢回登录页

**症状**: 输完密码点登录，跳到 dashboard 后立即跳回 login。

**原因**: 旧 cookie 用了旧 `JWT_SECRET` 签的，新 secret 验签失败 → 401 → 前端跳登录。

**解决**: F12 → Application → Cookies → `http://localhost:3000` → 删除 `booknook_token` → 重新登录。

### 7.9 PowerShell 拒绝执行 .ps1 脚本

**症状**: 跑 `.\scripts\00-init-db.ps1` 报 "无法加载文件 ... 因为禁止运行脚本"。

**解决**:
```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
# 输 Y 回车确认
```

只需开口一次，对当前用户永久生效。
