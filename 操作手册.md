## ✅ 3. 11 项基础功能 · 逐条演示路径

### 🔹 功能 1 · 用户管理 (PPT 1)

#### 1.1 登录
```
访问 http://localhost:3000  →  自动跳 /login
点击演示账号 super  →  自动填充用户名密码
点击 [登录] 按钮  →  跳转 /dashboard,顶部弹出"操作成功"toast
```
**助教可能追问**:
- Q: "密码是怎么加密的?"
  A: "我用 PPT 要求的 **MD5**,但加了 **12 字节随机 salt** 抵御彩虹表。每个用户独立 salt,即使同密码哈希也不同。**报告里也讨论了 MD5 的安全权衡,生产环境应换 bcrypt/argon2。**"
- Q: "登录态怎么存?"
  A: "JWT 放在 **HttpOnly Cookie** 里,不暴露给 JS,防 XSS;客户端还存了一份 token 在 localStorage,用于 Authorization 头兜底。"

#### 1.2 超管创建新用户 (右上侧栏: 用户管理)
```
进入 /users (仅超管菜单可见)
点击 [新建用户] 按钮  →  填写用户名/密码/姓名/工号/角色
点 [保存]  →  列表立刻出现新用户
```

#### 1.3 普通管理员只能改自己 (切到 admin1)
```
顶部头像 → [退出登录]  →  切 admin1 / Admin@2026  →  登录
观察侧栏:用户管理、操作日志菜单已消失
点击我的信息 → 可修改自己,但 /users 列表无法访问 (403)
```
**助教可能追问**:
- Q: "怎么实现的权限隔离?"
  A: "后端 `requireSuperAdmin` 中间件挂在 `/api/users` 和 `/api/logs` 之前,返回 403;前端侧栏根据 `useAuth().user.role` 条件渲染,**前后端双层防御**。"

## 🗄 5. 数据库讲解准备

### 5.1 ER 图 (主要表关系)

```
                ┌──────────────┐
                │    users     │ ◄────────────────────┐
                └──────┬───────┘                      │
                       │ 1 created_by                 │
              ┌────────┼────────┐                     │
              │ N      │ N      │ N                   │
              ▼        ▼        ▼                     │
   ┌──────────────┐ ┌──────────┐ ┌──────────────┐   │
   │ purchase_    │ │ sale_    │ │ transactions │   │
   │ orders       │ │ orders   │ │              │   │
   └────┬─────────┘ └────┬─────┘ └──────────────┘   │
        │ 1              │ 1                          │
        │ N              │ N                          │
        ▼                ▼                            │
   ┌──────────────┐ ┌──────────────┐                 │
   │ purchase_    │ │ sale_order_  │                 │
   │ order_items  │ │ items        │                 │
   └────┬─────────┘ └─────┬────────┘                 │
        │ N              │ N                          │
        └──┬─────────────┘                            │
           ▼                                          │
       ┌────────┐ ◄─ inventory_alerts                 │
       │ books  │                                     │
       └────────┘     operation_logs ─────────────────┘
```

### 5.2 范式证明 (BCNF 论证)

所有表均满足 **BCNF**:
- 每张表都存在 **id 主键** (代理键),保证 1NF (原子) + 2NF (无部分依赖)
- 非主属性不传递依赖 (例如 `subtotal` 是 STORED GENERATED,由 `quantity × unit_price` 计算,**不冗余存储**)
- 所有 FD 左侧均为超键 → BCNF

**唯一"看似冗余"** 的是 `transactions.amount` 与订单的 `total_amount`:
这是**事实表模式**,订单可改、流水不可改,符合财务系统设计准则,**报告中专门讨论**。

### 5.3 触发器一览

| 触发器 | 作用 |
|--------|------|
| `trg_sale_items_after_insert` | 销售 → 扣库存 + 检查库存预警 |
| `trg_sale_orders_after_insert` | 销售单 → 写入 income 流水 |
| `trg_purchase_status_change`  | 进货状态机:付款→写 expense,入库→更新 books |
| `trg_*_updated_at` × 3        | users/books/po 自动维护 `updated_at` |

### 5.4 关键索引

```sql
-- ILIKE '%xx%' 加速 (GIN 三元组)
CREATE INDEX idx_books_title_trgm     ON books USING gin (title     gin_trgm_ops);
CREATE INDEX idx_books_author_trgm    ON books USING gin (author    gin_trgm_ops);
CREATE INDEX idx_books_publisher_trgm ON books USING gin (publisher gin_trgm_ops);
-- 部分索引:只索引低库存
CREATE INDEX idx_books_low_stock      ON books (stock) WHERE stock <= low_stock_threshold;
-- 复合索引:进货单按状态+时间
CREATE INDEX idx_po_status_time       ON purchase_orders (status, created_at DESC);
```

## 🆘 8. 答辩 Q&A 预演 (10 个高频问题)

### Q1: 为什么用 Prisma + raw SQL 混合?
**A**: Prisma 提供类型安全的查询,90% 场景下高效;
但 `subtotal` 是 PostgreSQL 的 `GENERATED ALWAYS AS STORED` 列,Prisma 不支持写入,
所以 `purchase_order_items` / `sale_order_items` 的 INSERT 用 raw SQL —— **取长补短**。

### Q2: 为什么密码用 MD5 而不是 bcrypt?
**A**: PPT 明确要求 MD5。但我加了 **12 字节随机 salt**,且每个用户独立 salt,
极大提升了基础安全性,基本抵御彩虹表。**报告中我也专门讨论了为何生产环境应该升级到 bcrypt/argon2**,体现了对密码学的理解深度。

### Q3: 怎么保证销售扣库存的原子性?
**A**: 三层保护:
1. **数据库 CHECK 约束** `books.stock >= 0`,扣到负数直接报错
2. **Prisma 事务** `$transaction(async tx => {...})` 包裹整个结账
3. **PG 触发器** 在 `sale_order_items` 插入后立即更新 books,
   如失败,事务自动回滚整个销售单

### Q4: 触发器有什么风险?
**A**: 触发器隐藏了副作用,调试稍困难。我的应对:
- 触发器集中在一个 SQL 文件,加详细注释
- 业务文档明确写出每个触发器的作用 (见操作手册 §5.3)
- 关键状态变化时,业务代码会显式调用 (e.g. 进货付款先 UPDATE status,触发器再写流水)

### Q5: 为什么 transactions 不删除?
**A**: 财务流水是 **审计追溯证据**,符合企业财务系统设计准则 (sox/COSO)。
订单可以修改/删除,但每次操作都会在 transactions 留下不可篡改的记录。

### Q6: 为什么前后端真正分离?
**A**: 
- 体现 **PPT 推荐的工程能力**
- 后端可独立用 curl/Postman 测试,**降低答辩演示风险**
- 未来可前端换 Vue / 移动端,后端零改动
- API 边界明确,便于多人协作 (单人也保持工程纪律)

### Q7: 怎么防 SQL 注入?
**A**: 
- Prisma 全部参数化查询,**编译时类型检查**
- 极少数 raw SQL 用模板字符串 `prisma.$executeRaw\`INSERT ... ${val}\``,
  Prisma 会自动参数化,**不是字符串拼接**

### Q8: 为什么用 GIN 三元组索引?
**A**: 需求是 "模糊搜索书名/作者",对应 SQL `WHERE title ILIKE '%xx%'`。
B-tree 对前缀模糊有效,**对中间模糊无效**;GIN + pg_trgm 把字符串切成三元组,
即使 `%中%` 也是毫秒级。**索引选型思考**是数据库课的核心能力。

### Q9: 为什么图表数据走 view 而不是 ad-hoc query?
**A**: 
- **可维护性**:聚合 SQL 写在数据库一次,前端调多次
- **可优化**:DBA 看到 view 慢可加 materialized view
- **可重用**:报表导出和 Dashboard 共用同一份聚合逻辑

### Q10: 加分项里你最满意哪个?
**A**: 
- 创新:**操作日志中间件**,一个 30 行的 audit.ts 全局拦截所有写请求,JSONB 完整记录(密码字段自动脱敏)。**符合企业级 RBAC + 审计追溯要求**。
- 视觉:**暖书店主题**,从配色 (米色+琥珀) 到字体 (Playfair + Noto Serif SC) 到 Logo (CSS 绘制书架),**每个细节都呼应"图书"的主题**,不是套用通用模板。

## 9. 常见问题排查

### Q: 数据库初始化失败
```
[ERROR] FATAL:  password authentication failed
```
**解决**: PowerShell 中先设置 `$env:PGPASSWORD = "你的密码"`,再跑脚本。

### Q: 前端报 `ECONNREFUSED localhost:4000`
**解决**: 后端没启动,先 `cd backend && npm run dev` 等到看见 `🌿 BookNook Backend ready`。

### Q: 浏览器一直白屏
**解决**:
1. 打开 F12 控制台看红色错误
2. 看后端终端是否有 5xx 错误
3. 数据库是否真在跑: `psql -U postgres -d booknook -c "SELECT 1"`

### Q: 端口冲突 (3000 或 4000 被占)
**解决**:
```powershell
# Windows: 找出占用端口的进程
netstat -ano | findstr :3000
taskkill /PID <PID> /F
```
或在 `backend/.env` 改 `PORT=4001`,在 `frontend/next.config.mjs` 改 `BACKEND_URL`。

### Q: PDF 中文显示方框
**解决**: PDF 字体功能为 nice-to-have,Excel 才是主报表;
若要 PDF 中文,放一份 `NotoSerifSC-Regular.ttf` 到 `backend/assets/fonts/`。
**演示时优先用 Excel 即可,效果更好**。

### Q: Prisma 报 `Native binding`
**解决**: 杀掉所有 node 进程,删 `backend/node_modules/.prisma`,
重跑 `npx prisma generate`。
