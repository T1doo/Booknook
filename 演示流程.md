> 每一步给出 **① 做什么 · ② 关键代码 · ③ 要对助教讲的点**。

> **三个窗口**：① 数据库（一次性初始化） ② 后端 `:4000` ③ 前端 `:3000`。

## 1. 初始化数据库

**做什么**：跑一键脚本，重建 `booknook` 库并顺序执行 schema / seed / views 三段 SQL。

```powershell
$env:PGPASSWORD = "060425"
.\scripts\00-init-db.ps1
# 成功末尾: [DONE] BookNook 数据库初始化完毕!
```

脚本核心（`scripts/00-init-db.ps1`）：

```powershell
# WITH (FORCE) 自动踢掉后端 Prisma 的活跃连接 (PG13+)，重置时不必先停后端
& $psql @baseArgs -c "DROP DATABASE IF EXISTS $DbName WITH (FORCE);"
& $psql @baseArgs -c "CREATE DATABASE $DbName ENCODING 'UTF8' LC_COLLATE 'C' LC_CTYPE 'C' TEMPLATE template0;"
# 顺序执行三段脚本，任一报错立即停 (ON_ERROR_STOP=1)
& $psql @dbArgs -f 01-schema.sql   # 建结构
& $psql @dbArgs -f 02-seed.sql     # 灌数据
& $psql @dbArgs -f 03-views.sql    # 建视图
```

**讲点**：
- 三段式拆分：`01-schema`（结构）/ `02-seed`（数据）/ `03-views`（视图），职责清晰、可单独重跑。
- `WITH (FORCE)`：重置时自动断开后端连接，演示时不用先关后端。
- `ENCODING UTF8 / LC_COLLATE C`：统一编码，避免中文乱码与排序歧义。
- `ON_ERROR_STOP=1`：SQL 一出错就中断，不会带着半截数据继续往下走。

## 2. 建表与约束（`database/01-schema.sql`）

**做什么**：建 4 个枚举 + 10 张表，约束尽量下沉到表定义里。

枚举与扩展：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;   -- GIN 三元组索引，加速 ILIKE '%关键字%'
CREATE EXTENSION IF NOT EXISTS pgcrypto;  -- 随机数（生成 salt）

CREATE TYPE user_role        AS ENUM ('super_admin', 'admin');
CREATE TYPE purchase_status  AS ENUM ('pending', 'paid', 'returned', 'received');
CREATE TYPE transaction_type AS ENUM ('income', 'expense');
```

用户表（密码 MD5+salt）：

```sql
CREATE TABLE public.users (
    id              BIGSERIAL PRIMARY KEY,
    username        VARCHAR(32)  NOT NULL UNIQUE,
    password_hash   CHAR(32)     NOT NULL,                 -- md5 输出固定 32 hex
    salt            CHAR(24)     NOT NULL,                 -- 12 字节 -> 24 hex
    real_name       VARCHAR(32)  NOT NULL,
    employee_no     VARCHAR(16)  NOT NULL UNIQUE,
    gender          gender_type  NOT NULL DEFAULT 'other',
    age             SMALLINT     CHECK (age IS NULL OR (age BETWEEN 16 AND 100)),
    role            user_role    NOT NULL DEFAULT 'admin',
    is_active       BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

库存书籍表（GIN 模糊搜索 + 库存 CHECK）：

```sql
CREATE TABLE public.books (
    id                   BIGSERIAL PRIMARY KEY,
    isbn                 VARCHAR(20)   NOT NULL UNIQUE,
    title                VARCHAR(200)  NOT NULL,
    publisher            VARCHAR(100)  NOT NULL,
    author               VARCHAR(100)  NOT NULL,
    retail_price         NUMERIC(10,2) NOT NULL CHECK (retail_price >= 0),
    stock                INTEGER       NOT NULL DEFAULT 0 CHECK (stock >= 0),
    low_stock_threshold  INTEGER       NOT NULL DEFAULT 5 CHECK (low_stock_threshold >= 0),
    category             VARCHAR(50),
    created_at/updated_at TIMESTAMPTZ ...
);
CREATE INDEX idx_books_title_trgm  ON public.books USING gin (title  gin_trgm_ops);
CREATE INDEX idx_books_author_trgm ON public.books USING gin (author gin_trgm_ops);
```

进货单 + 明细（状态机字段 + 生成列）：

```sql
CREATE TABLE public.purchase_orders (
    id           BIGSERIAL PRIMARY KEY,
    order_no     VARCHAR(32)     NOT NULL UNIQUE,
    created_by   BIGINT          NOT NULL REFERENCES public.users(id),
    status       purchase_status NOT NULL DEFAULT 'pending',
    total_amount NUMERIC(12,2)   NOT NULL DEFAULT 0 CHECK (total_amount >= 0),
    paid_at      TIMESTAMPTZ, returned_at TIMESTAMPTZ, received_at TIMESTAMPTZ, ...
);

CREATE TABLE public.purchase_order_items (
    id             BIGSERIAL PRIMARY KEY,
    order_id       BIGINT NOT NULL REFERENCES public.purchase_orders(id) ON DELETE CASCADE,
    book_id        BIGINT REFERENCES public.books(id),    -- 新书首次进货时为 NULL，入库时回填
    isbn/title/publisher/author ...,
    purchase_price NUMERIC(10,2) NOT NULL CHECK (purchase_price >= 0),
    quantity       INTEGER       NOT NULL CHECK (quantity > 0),
    retail_price   NUMERIC(10,2) CHECK (retail_price IS NULL OR retail_price >= 0),
    subtotal       NUMERIC(12,2) GENERATED ALWAYS AS (purchase_price * quantity) STORED
);
```

销售明细 + 财务流水 + 库存预警（关键约束）：

```sql
CREATE TABLE public.sale_order_items (
    order_id   BIGINT NOT NULL REFERENCES public.sale_orders(id) ON DELETE CASCADE,
    book_id    BIGINT NOT NULL REFERENCES public.books(id),
    quantity   INTEGER       NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(10,2) NOT NULL CHECK (unit_price >= 0),
    subtotal   NUMERIC(12,2) GENERATED ALWAYS AS (unit_price * quantity) STORED
);

CREATE TABLE public.transactions (              -- 不可篡改的流水账（事实表）
    type          transaction_type NOT NULL,
    amount        NUMERIC(12,2)    NOT NULL CHECK (amount > 0),
    related_table VARCHAR(32)      CHECK (related_table IN ('purchase_orders','sale_orders')),
    related_id    BIGINT,
    CHECK (related_id IS NULL OR related_table IS NOT NULL)
);

-- 同一本书最多只能有一条未解决预警（并发安全）
CREATE UNIQUE INDEX uq_alerts_book_open ON public.inventory_alerts (book_id) WHERE resolved = FALSE;
```

其余表一句话：`sale_orders`（销售单头）、`operation_logs`（操作日志，JSONB payload）、`i18n_dict`（中英词条）。

**讲点**：
- 全部表满足 **BCNF**，统一用 `BIGSERIAL` 代理主键。
- `subtotal` 用 `GENERATED ALWAYS AS ... STORED`：金额由库计算，不冗余、永远和单价×数量一致。
- 约束下沉：`CHECK (stock >= 0)` 是防超卖的最后一道兜底；`amount > 0`、`related_table` 取值受限。
- 模糊搜索建 **GIN 三元组索引**（`pg_trgm`），`ILIKE '%中%'` 也能走索引（B-tree 对中间模糊无效）。
- `inventory_alerts` 用**部分唯一索引** `WHERE resolved=FALSE`，保证一本书同时只有一条未解决预警。
- 进货明细 `book_id` 可空：新书首次进货时 books 里还没有，入库时才回填。

## 3. 触发器与函数（核心 · 同 `01-schema.sql`）

**做什么**：把业务规则封装进数据库——应用层只发简单 SQL，规则绕不开。

订单号生成（并发安全）：

```sql
CREATE OR REPLACE FUNCTION gen_order_no(prefix TEXT) RETURNS TEXT AS $$
DECLARE today TEXT := to_char(now(),'YYYYMMDD'); seq INT;
        lock_key BIGINT := hashtextextended(prefix || today, 0);
BEGIN
    PERFORM pg_advisory_xact_lock(lock_key);           -- 同日同前缀串行化
    SELECT COALESCE(MAX(...),0)+1 INTO seq FROM ... WHERE order_no LIKE prefix||'-'||today||'-%';
    RETURN prefix||'-'||today||'-'||lpad(seq::TEXT,4,'0');   -- 形如 PO-20260528-0001
END $$ LANGUAGE plpgsql;
```

销售：行级扣库存 + 语句级写流水（两个触发器）：

```sql
-- 行级：每条明细插入后扣库存 + 触发预警检查
CREATE FUNCTION fn_sales_after_insert() RETURNS TRIGGER AS $$ BEGIN
    UPDATE public.books SET stock = stock - NEW.quantity WHERE id = NEW.book_id;
    IF NOT FOUND THEN RAISE EXCEPTION 'Book id=% 不存在', NEW.book_id; END IF;
    PERFORM fn_books_low_stock_check(NEW.book_id);
    RETURN NEW; END $$ LANGUAGE plpgsql;
CREATE TRIGGER trg_sale_items_after_insert
    AFTER INSERT ON sale_order_items FOR EACH ROW EXECUTE FUNCTION fn_sales_after_insert();

-- 语句级：用本次 statement 的明细 SUM(subtotal) 写一条 income 流水
CREATE FUNCTION fn_sale_items_after_insert_stmt() RETURNS TRIGGER AS $$ BEGIN
    INSERT INTO public.transactions (type, amount, related_table, related_id, description, created_by)
    SELECT 'income', SUM(soi.subtotal), 'sale_orders', so.id, '销售单 '||so.order_no, so.created_by
    FROM (SELECT DISTINCT order_id FROM new_items) ni
    JOIN sale_orders so ON so.id = ni.order_id
    JOIN sale_order_items soi ON soi.order_id = so.id
    GROUP BY so.id, so.order_no, so.created_by;
    RETURN NULL; END $$ LANGUAGE plpgsql;
CREATE TRIGGER trg_sale_items_after_insert_stmt
    AFTER INSERT ON sale_order_items
    REFERENCING NEW TABLE AS new_items                 -- 关键：拿到本次插入的全部新行
    FOR EACH STATEMENT EXECUTE FUNCTION fn_sale_items_after_insert_stmt();
```

进货状态机（合法转移 + 入库回填，其余 `RAISE EXCEPTION`）：

```sql
CREATE FUNCTION fn_purchase_status_change() RETURNS TRIGGER AS $$ BEGIN
    IF OLD.status='pending' AND NEW.status='paid' THEN          -- 付款：写支出流水
        NEW.paid_at := now();
        INSERT INTO transactions(type,amount,related_table,related_id,description,created_by)
        VALUES ('expense', NEW.total_amount, 'purchase_orders', NEW.id, '进货单 '||NEW.order_no||' 付款', NEW.created_by);
    ELSIF OLD.status='pending' AND NEW.status='returned' THEN   -- 退货：仅打标记
        NEW.returned_at := now();
    ELSIF OLD.status='paid' AND NEW.status='received' THEN      -- 入库：新书 INSERT / 老书 +stock
        FOR item IN SELECT * FROM purchase_order_items WHERE order_id=NEW.id LOOP
            -- 新书 -> INSERT books；老书 -> stock+qty, retail_price=COALESCE(item.retail_price, retail_price)
            -- ISBN 命中老书但书名/作者/出版社不一致 -> RAISE EXCEPTION（防"老书改名"）；并回填 book_id
        END LOOP;
    ELSE
        RAISE EXCEPTION '非法的进货单状态变更: % -> %', OLD.status, NEW.status;
    END IF;
    RETURN NEW; END $$ LANGUAGE plpgsql;
CREATE TRIGGER trg_purchase_status_change
    BEFORE UPDATE OF status ON purchase_orders
    FOR EACH ROW WHEN (OLD.status IS DISTINCT FROM NEW.status)
    EXECUTE FUNCTION fn_purchase_status_change();
```

**讲点**：
- **业务规则封装在 DB**：付款写支出、入库改库存、销售扣库存写收入，全部触发器驱动；应用代码只发一句 `UPDATE status` 或 `INSERT items`，连 psql 直连改库也绕不开。
- **状态机硬约束**：只允许 `pending→paid / pending→returned / paid→received`，其它一律 `RAISE EXCEPTION`。
- **流水金额由 DB 强制 = 明细 SUM**：语句级触发器用 `REFERENCING NEW TABLE` 取本次插入的全部明细求和，前端传的 `total_amount` 篡改无效。
- **并发安全**：`gen_order_no` 用 `pg_advisory_xact_lock` 把「同日同前缀」串行化，避免订单号撞 UNIQUE。
- **入库一致性**：老书价用 `COALESCE` 默认沿用；ISBN 命中但元数据不一致直接报错。

## 5. 创建视图（`database/03-views.sql`）

**做什么**：为 Dashboard / 报表导出提供 6 个聚合视图。

```sql
CREATE VIEW v_monthly_finance AS
SELECT to_char(created_at,'YYYY-MM') AS month,
       SUM(CASE WHEN type='income'  THEN amount ELSE 0 END)       AS income,
       SUM(CASE WHEN type='expense' THEN amount ELSE 0 END)       AS expense,
       SUM(CASE WHEN type='income'  THEN amount ELSE -amount END) AS net
FROM public.transactions GROUP BY 1 ORDER BY 1;
```

其余视图：`v_book_sales_summary`（Top10 畅销）、`v_daily_sales_trend`（近 30 天，用独立子查询避免 JOIN 放大金额）、`v_low_stock_books`、`v_user_activity`、`v_purchase_pipeline`。

**讲点**：
- 聚合逻辑写在库里一次，Dashboard 和报表导出共用，可维护、可复用。
- 以后视图慢了可平滑升级成 materialized view，业务代码不动。
- `v_daily_sales_trend` 故意用三个独立子查询，避免「一单多项」时 JOIN 把 `total_amount` 按行数放大。

## 8. 登录鉴权（`backend/src/modules/auth/auth.controller.ts`）

**做什么**：校验 MD5+salt 密码 → 签 JWT → 写 HttpOnly Cookie。

```ts
const user = await prisma.user.findUnique({ where: { username } });
if (!user || !user.is_active) throw Err.unauthorized('用户名或密码错误');
if (!verifyPassword(password, user.salt, user.password_hash)) throw Err.unauthorized('用户名或密码错误');

const token = signToken({ uid: user.id.toString(), username: user.username, role: user.role });
res.cookie(env.COOKIE_NAME, token, {
  httpOnly: true, secure: !env.isDev, sameSite: 'strict', path: '/api',
});
```

密码与 JWT 实现（`utils/md5.ts` / `utils/jwt.ts`）：

```ts
// 每用户独立 salt；常量时间比较防时序攻击
export function verifyPassword(pwd, salt, expected) {
  const actual = md5Hash(pwd, salt);                        // md5(pwd || salt)
  return timingSafeEqual(Buffer.from(actual,'hex'), Buffer.from(expected,'hex'));
}
// 算法显式锁 HS256，防 alg=none / RS-HS 混淆伪造
jwt.sign(payload, env.JWT_SECRET, { algorithm: 'HS256', expiresIn: '7d' });
```

登录还挂了速率限制：15 分钟内同一 IP 最多 20 次（`express-rate-limit`）。

**讲点**：
- PPT 要求 MD5，我**加了每用户独立 12 字节 salt** 抵御彩虹表；比对用 `timingSafeEqual` 常量时间，防时序攻击。
- token 放 **HttpOnly Cookie**，JS 读不到（XSS 偷不走）；`sameSite=strict` + `path=/api` 收窄 CSRF。
- JWT 显式锁 **HS256**，防 `alg=none` 伪造；登录限流抬高字典攻击成本。

## 9. 库存查询与修改（`backend/src/modules/books/books.controller.ts`）

**做什么**：多字段模糊查询；改书价时留 before/after 审计。

```ts
// 多字段查询：all / id / isbn / title / author / publisher；'all'+纯数字还兼顾按 ID
where.OR = [
  { isbn:   { equals: kw } },
  { title:  { contains: kw, mode: 'insensitive' } },   // -> ILIKE，命中 GIN 索引
  { author: { contains: kw, mode: 'insensitive' } },
  { publisher: { contains: kw, mode: 'insensitive' } },
  ...(/^\d+$/.test(kw) ? [{ id: BigInt(kw) }] : []),
];
```

```ts
// PATCH /books/:id：先把原值快照塞进 req，由 audit 中间件合并写 before/after
req.auditBefore = { title: book.title, author: book.author, publisher: book.publisher,
                    retail_price: book.retail_price.toString(), category: book.category };
const updated = await prisma.book.update({ where: { id }, data: req.body });
```

**讲点**：
- `contains + insensitive` 被 Prisma 编译成 `ILIKE '%kw%'`，命中第 2 步建的 GIN 三元组索引，百万级也是毫秒级。
- 删除有保护：库存 > 0 或已有销售记录则拒绝。
- 改书价是关键业务字段，controller 埋 `auditBefore`，审计中间件自动留「改前/改后」对照。

## 10. 进货全生命周期（`backend/src/modules/purchases/purchases.controller.ts`）

**做什么**：建单（pending）→ 付款 / 退货 / 入库；状态推进用原子 `updateMany`。

```ts
// 创建：订单号由 DB 函数生成；明细用 raw SQL 插（subtotal 是 GENERATED 列，Prisma 不能写）
const [{ order_no }] = await tx.$queryRaw`SELECT gen_order_no('PO') AS order_no`;
await tx.$executeRaw`INSERT INTO purchase_order_items (...) VALUES (...)`;
```

```ts
// 状态推进：把"读 + 写"合并成一条带行锁的原子 SQL，避免并发双写流水
async function transitStatus(id, from, to) {
  const result = await prisma.purchaseOrder.updateMany({
    where: { id, status: { in: from } }, data: { status: to },
  });
  if (result.count === 0) {                  // 没改到：不存在 / 已是目标态(幂等) / 状态不允许
    const po = await prisma.purchaseOrder.findUnique({ where: { id } });
    if (po?.status === to) return po;        // 幂等返回
    throw Err.badRequest(`当前状态 ${po.status}，不允许变更为 ${to}`);
  }
}
```

**讲点**：
- `updateMany(where:{id, status:{in:from}})` 把「检查当前状态 + 改状态」合并成一条 SQL，DB 加行锁；两个并发付款只有一个 `count===1` 成功，另一个幂等返回，**触发器只触发一次、流水只写一条**。
- 真正的副作用（写支出 / 入库改库存 / 元数据校验）都在第 3 步的状态机触发器里，controller 很薄。
- 入库时新书必填零售价、老书 `COALESCE` 沿用——UI 与触发器两层契约一致。

## 11. 销售收银（**答辩重点** · `backend/src/modules/sales/sales.controller.ts`）

**做什么**：一个事务里行锁校验库存 → 建单 → 批量插明细（触发器扣库存写流水）→ 应用层对账。

```ts
const so = await prisma.$transaction(async (tx) => {
  // ① 行锁：锁住本次涉及的所有 books 行，让并发结账串行化
  const locked = await tx.$queryRaw`
    SELECT id, title, stock, retail_price FROM books
     WHERE id IN (${Prisma.join(bookIds)}) FOR UPDATE`;

  // ② 校验库存 + 按"分"为整数累加金额，避免 JS Number 浮点误差
  for (const it of body.items) {
    if (b.stock < it.quantity) throw Err.badRequest(`《${b.title}》库存不足 (当前 ${b.stock})`);
    totalCents += BigInt(Math.round(Number(b.retail_price)*100)) * BigInt(it.quantity);
  }

  // ③ 建单 + 一次 statement 批量插明细（配合语句级触发器写收入流水）
  const [{ order_no }] = await tx.$queryRaw`SELECT gen_order_no('SO') AS order_no`;
  const order = await tx.saleOrder.create({ data: { order_no, total_amount: total, ... } });
  await tx.$executeRaw(Prisma.sql`INSERT INTO sale_order_items (...) VALUES ${Prisma.join(valuesSql)}`);

  // ④ 对账：items SUM 必须等于 total，否则整单回滚
  const [{ sum }] = await tx.$queryRaw`SELECT COALESCE(SUM(subtotal),0)::TEXT AS sum
                                         FROM sale_order_items WHERE order_id=${order.id}`;
  if (sum !== total) throw Err.internal('对账失败');
  return order;
});
```

**讲点**：
- **三层防超卖**：① 应用层 `SELECT ... FOR UPDATE` 行锁让并发结账串行；② 触发器 `UPDATE stock = stock - qty` 实扣；③ 表上 `CHECK (stock>=0)` 兜底，前两层都失效也改不成负数。
- 明细**一次 statement 批量插**，正好配第 3 步语句级触发器按 `SUM(subtotal)` 写一条收入流水。
- 金额按「分」整数累加避开浮点误差；事务结束前再用 SQL 对账一次，不一致整单回滚。
- 全程 Prisma 参数化（`Prisma.join` / `Prisma.Decimal`），无字符串拼接，防 SQL 注入。

## 12. 财务流水（`backend/src/modules/transactions/transactions.controller.ts`）

**做什么**：按类型 / 日期筛选 + 收支汇总，**只读**。

```ts
if (q.type) where.type = q.type;
if (q.from || q.to) where.created_at = { gte: from?, lte: to? };
const [total, list, agg] = await Promise.all([
  prisma.transaction.count({ where }),
  prisma.transaction.findMany({ where, orderBy: { created_at: 'desc' }, ... }),
  prisma.transaction.groupBy({ by: ['type'], where, _sum: { amount: true } }),  // 收支汇总
]);
```

**讲点**：
- `transactions` 是**事实表 / 流水账**：应用层只 `SELECT`，从不 `UPDATE/DELETE`；所有写入都由触发器驱动。
- 每条流水带 `related_table + related_id`，可反查到对应进货单 / 销售单。
- 走 `idx_tx_type_time` 索引，类型 + 时间范围筛选高效。

## 13. 加分项串讲

RBAC 三层防御（`backend/src/middlewares/rbac.ts`）：

```ts
export const requireSuperAdmin = requireRole('super_admin');
// users / logs 路由内：router.use(authRequired, requireSuperAdmin)
```

- ① 后端中间件：普通管理员访问 `/api/users`、`/api/logs` 返回 403；② 前端 `RoleGuard` 自动跳回 dashboard；③ 侧栏按角色条件渲染。前端两层是体验，后端一层是安全底线，绕过前端照样 403。

操作日志审计（`backend/src/middlewares/audit.ts`，零侵入）：

```ts
res.on('finish', () => {
  if (res.statusCode >= 400) return;                       // 只记成功操作
  const payload = auditBefore ? { before, after } : after;
  prisma.operationLog.create({ data: { user_id, action, resource, ip, payload } })
    .catch(...);                                           // 不 await，不阻塞主流程
});
const SENSITIVE_RE = /password|secret|token|api[_-]?key|.../i;   // 深度递归脱敏为 ***
```

其余：Dashboard 直接读第 5 步的视图（前端 Recharts 渲染）；报表用 `exceljs` / `pdfkit` 流式导出 XLSX/PDF；中英 i18n + 暗色主题。

**讲点**：
- 审计中间件全局挂在写请求上，业务代码完全不感知；密码等字段**深度递归脱敏**为 `***`。
- 全局写操作的审计 + 三层 RBAC，体现企业级权限与追溯思路。
